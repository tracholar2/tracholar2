为什么要使用Expression Func T 而不是Func T？

By zhizi

Published 2018-03-17

Contents

我理解lambdas和Func和Action委托。但是表达式让我感到遗憾。在什么情况下，您会使用Expression＆lt; Func＆lt; T＆gt;＆gt;而不是普通的旧的Func＆lt; T＆gt;？

当你想把lambda表达式当作表达式树来看，而不是执行它们。例如，LINQ to
SQL获取表达式并将其转换为等效的SQL语句，并将其提交给服务器（而不是执行lambda）。

从概念上讲，Expression＆lt; Func＆lt; T＆gt;＆lt; / code＆gt;与 Func＆lt; T＆gt; 完全不同。 Func＆lt; T＆gt;表示一个委托，它几乎是指向一个方法的指针，而＆lt; code＆gt; Expression＆lt; Func＆lt;
T＆gt;＆lt; / code＆gt;表示一个 >树形数据结构 为lambda表达式。这个树结构 描述了lambda表达式的作用
，而不是做实际的事情。它基本上保存有关表达式，变量，方法调用，…的组成的数据（例如，它保存诸如这个lambda的信息是某个常量+某个参数）。你可以使用这个描述把它转换成一个实际的方法（用Expression.Compile）或者用其他的东西（比如LINQ to
SQL的例子）。把lambda作为匿名方法和表达式树对待纯粹是一个编译时间的事情。

Func<int> myFunc = () => 10; // similar to: int myAnonMethod() { return 10; }

将有效地编译成一个没有任何东西并返回10的IL方法。

Expression<Func<int>> myExpression = () => 10;

将被转换为描述一个表达式的数据结构，该表达式不会得到参数并返回值10：

虽然在编译时它们看起来是一样的，但编译器产生的却是 完全不同的 。

我加了一个答案，因为这些答案似乎凌驾于我的头上，直到我意识到它有多简单。有时候，你的期望是复杂的，使得你无法围绕它“

”

直到我走进一个非常恼人的“bug”，试图一般地使用LINQ到SQL：我不需要了解它们之间的区别。

public IEnumerable<T> Get(Func<T, bool> conditionLambda){
  using(var db = new DbContext()){
    return db.Set<T>.Where(conditionLambda);
  }
}

这很好，直到我开始在较大的数据集上发生OutofMemoryExceptions。在lambda里面设置断点让我意识到，它是遍历我的表中的每一行一个接一个寻找匹配我的lambda条件。这难倒了我一段时间，因为为什么它是对待我的数据表作为一个巨大的IEnumerable而不是做LINQ到SQL像它应该？在我的LINQ到MongoDb版本中，它也完全一样。

解决方法是将Func＆lt; T，bool＆gt;转换成Expression＆lt; Func＆lt; T，bool＆gt;＆gt;，所以我去Google为什么需要一个表达式，而不是Func，结束于此。

表达式只是简单地将委托转换为关于自己的数据。 所以a =＆gt; a + 1就像“在左边有一个int a，在右边你加1”。
就是这样。 你现在可以回家了。这显然比这更结构化，但这基本上都是一个表达树 - 没有任何东西可以包裹你的头。

理解这一点，很明显为什么LINQ-to-SQL需要一个Expression，而一个Func是不够的。 Func并没有提供一种进入自身的方法，以查看如何将其转换为SQL / MongoDb
/其他查询的细节。你不能看到是否加减乘法。你所能做的就是运行它。另一方面，表达式允许您查看委托内部并查看它想要执行的任何操作，从而使您能够将其转换为所需的任何内容，如SQL查询。 Func
`不起作用，因为我的DbContext对于实际在lambda表达式中将其转换为SQL的内容是盲目的，所以它做了下一个最好的事情，并通过表中的每一行迭代该条件。
/ p> 编辑：在约翰·彼得的请求上阐述我的最后一句话：

IQueryable扩展IEnumerable，因此IEnumerable的方法（如Where（））获取接受Expression的重载。当你将一个Expression传递给它的时候，你保留了一个IQueryable，但是当你传递一个Func的时候，你将回退到基本的IEnumerable，得到一个IEnumerable结果。换句话说，没有注意到你已经把你的数据集变成了一个被迭代的列表，而不是被查询的东西。直到你真正看到签名的底层，很难发现有什么不同。
