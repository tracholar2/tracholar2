为什么不从List T继承？

在规划我的计划时，我经常从一个如此的思路开始：

A football team is just a list of football players. Therefore, I should
represent it with:

var football_team = new List<FootballPlayer>();

The ordering of this list represent the order in which the players are
listed in the roster.

但是我后来意识到，除了单纯的球员名单之外，球队还有其他属性，必须记录下来。例如，本赛季得分总数，当前预算，统一颜色，表示球队名称的string等等。

那么我想：

Okay, a football team is just like a list of players, but additionally, it
has a name (a string) and a running total of scores (an int). .NET does
not provide a class for storing football teams, so I will make my own class.
The most similar and relevant existing structure is List<FootballPlayer>, so
I will inherit from it:

class FootballTeam : List<FootballPlayer>

{

    public string TeamName;

    public int RunningTotal

}

但事实证明，[指南说，你不应该继承List＆lt; T＆gt; 。我在这方面完全被两方面弄糊涂了。

Why not?

很显然， List是为了性能而优化的。怎么会这样？如果扩展List，会导致哪些性能问题？什么会打破？

我见过的另外一个原因是，List是由微软提供的，我无法控制它，所以[。但我很难理解这一点。什么是公共API，我为什么要关心？如果我目前的项目没有，也不可能拥有这个公共API，我可以放心地忽略这个指南吗？如果我从List 和 继承，事实证明我需要一个公共API，那么我会遇到什么困难？

为什么它甚至重要？列表是一个列表。什么可能改变？我可能想要改变什么？

最后，如果微软不希望我从List继承，为什么他们没有把类封装成？

What else am I supposed to use?

显然，对于自定义集合，Microsoft提供了一个Collection类，应该扩展而不是List。但是这个类是非常光鲜的，并没有很多有用的东西，比如AddRange < >，例如。 jvitor83的答案提供了该特定方法的性能基本原理，但是如何缓慢AddRange不是更好比没有AddRange？

继承自Collection比从List继承更有效，我看不出有什么好处。毫无疑问，微软不会让我无缘无故地做额外的工作，所以我不禁感觉到我在某种程度上误解了某些东西，继承Collection其实并不是我的问题的正确解决方案。 / p>

我见过一些建议，比如实现IList。就是不行。这是几十行样板代码，没有任何收获。

最后，有人建议将List包装在

中

class FootballTeam 
{ 
    public List<FootballPlayer> Players; 
}

这有两个问题：

  • 它使我的代码不必要地冗长。我现在必须调用my_team.Players.Count而不是my_team.Count。谢天谢地，用C＃我可以定义索引器来使索引透明，并且转发内部的List的所有方法…但是这是很多代码！我为什么得到这些工作？

  • 这显然没有任何意义。足球队没有“拥有”球员名单。它是_球员的名单。你不会说“John McFootballer加入了SomeTeam的球员”。你说“John加入了SomeTeam”。您不要将字母添加到“字符串的字符”，您将字母添加到字符串。你不会把一本书添加到图书馆的书籍中，而是将一本书添加到一个图书馆。

我意识到，“引擎盖下”发生的事情可以被认为是“把X添加到Y的内部列表”，但这似乎是一种非常直观的思考世界的方式。

My question (summarized)

代表一个数据结构的正确的C＃方式是什么？“逻辑上”（也就是说，“对于人类的思维”）只是things的list / code>几个花里胡哨的？

从List＆lt; T＆gt;继承永远不可接受？什么时候可以接受？为什么/为什么不呢？程序员在决定是否从List＆lt; T＆gt;继承时，必须考虑什么？

这里有一些很好的答案。我会给他们添加以下几点。

What is the correct C# way of representing a data structure, which,
“logically” (that is to say, “to the human mind”) is just a list of things
with a few bells and whistles?

问任何十位熟悉足球存在的非计算机程序员的人填补空白：

A football team is a particular kind of _____

是否说了一些“花哨的足球运动员名单”，还是都说“运动队”或“俱乐部”或“组织”？你的观点是，一个足球队是一种特殊的球员列表_是在你的人的头脑和你的人的头脑中。

List＆lt; T＆gt;是一个 机制 。足球队是一个 业务对象 - 也就是表示某个概念的对象，该概念位于程序的 业务域
中。不要混合这些！一支足球队是一种团队，它有一个花名册，一个花名册 是一个球员列表 。名册不是一个特殊类型的球员列表。名册 是
球员列表。因此，创建一个名为Roster的List＆lt; Player＆gt;属性。除非你相信每个了解足球队的人都会从队伍中删除球员，否则你可以在ReadOnlyList＆lt; Player＆gt;

Is inheriting from List<T> always unacceptable?

谁不能接受？我？否。

When is it acceptable?

在构建 扩展List＆lt; T＆gt;机制 的机制时。

What must a programmer consider, when deciding whether to inherit from
List<T> or not?

我是建立一个 机制 还是一个 业务对象 ？

But that’s a lot of code! What do I get for all that work?

你花了更多的时间来输入你的问题，它会让你为List＆lt; T＆gt;的相关成员编写50次转发方法。你显然不怕冗长，我们在这里只是谈论很少的代码。这是几分钟的工作。

UPDATE

我给了它更多的思考，还有一个原因，就是不能把一个足球队列为一个球员列表。事实上，把一个足球队建模成拥有一个队员名单也许是个不错的主意。一个球队的问题在于你所拥有的是一个
快照 的团队 在某个时刻
。我不知道你们这个班的商业案例是什么，但是如果我有一个代表一个足球队的课，我想问一个问题，比如“2003年到2013年有多少海鹰队员因伤缺席比赛？或者“之前效力于另外一支球队的丹佛球员的年增长率是多少？”或“猪人今年一路走来？”

也就是说，足球队在我看来很好的模仿了一系列历史事实，比如当一名球员被招募，受伤，退役等等。显然目前的球员名册是一个重要的事实，可能应该是前面和中心，但是可能还有其他有趣的事情，你想要做这个对象，需要更多的历史观点。

Lastly, some suggest wrapping the List in something:

这是正确的方法。 “不必要的罗嗦”是看这个不好的方法。当你编写my_team.Players.Count时，它有明确的含义。你想统计玩家。

my_team.Count

意味着什么。算什么？

一个球队不是一个名单 - 不仅仅是一个球员名单。一个球队拥有球员，所以球员应该是其中的一员（一名成员）。

如果你真的担心它过于冗长，你可以随时暴露团队的属性：

public int PlayerCount {
    get {
        return Players.Count;
    }
}

..变成：

my_team.PlayerCount

现在这有意义，并遵守德米特法则。

您还应该考虑遵守复合重用原则。通过继承List＆lt; T＆gt;，你就是说一个团队 **是一个玩家列表，并且暴露出不必要的方法。这是不正确的 -
正如你所说，一个团队不仅仅是一个球员名单：它有一个名字，经理，董事会成员，培训师，医务人员，工资帽等等。通过让你的团队类别包含一个球员名单，你“一个队伍有一个队员名单”，但也可以有其他的一些东西。
