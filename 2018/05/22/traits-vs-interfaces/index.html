性状与接口

By zhizi

Published 2018-05-22

Contents
  1. 1. Interfaces
  2. 2. Traits

最近我一直在试图研究PHP，我发现自己陷入了性格上。我理解横向代码重用的概念，不想从抽象类继承。我不明白的是使用特征和接口之间的关键区别是什么？

我尝试过寻找一篇像样的博客文章或文章来解释何时使用其中一个，但我迄今发现的例子看起来非常相似以至于完全相同。

任何人都可以在这里分享他们的意见/观点吗？

一个接口定义了实现类 必须 实现的一组方法。

当一个特征是use‘d时，方法的实现也会出现 - 这不会发生在Interface中。

这是最大的区别。

来自水平重复使用PHP RFC ：

Traits is a mechanism for code reuse in single inheritance languages such as
PHP. A Trait is intended to reduce some limitations of single inheritance by
enabling a developer to reuse sets of methods freely in several independent
classes living in different class hierarchies.

公共服务公告：**

我想陈述一下，我相信特质几乎总是一种代码味道，应该避免使用构图。我认为单一继承经常被滥用到反模式的地步，而多重继承只会加剧这个问题。在大多数情况下，通过支持合成而不是继承（无论是单个还是多个），你会得到更好的服务。如果您仍然对特质及其与接口的关系感兴趣，请继续阅读…

先说这个：

Object-Oriented Programming (OOP) can be a difficult paradigm to grasp. Just
because you’re using classes doesn’t mean your code is Object-Oriented (OO).

要写OO代码，你需要明白OOP是关于你的对象的能力的。你必须考虑他们 可以做什么 ，而不是 实际做什么
。这与传统的程序编程形成了鲜明的对比，其中重点是编写一些代码“做某事”。

如果OOP代码是关于规划和设计的，那么界面就是蓝图，一个对象就是完全构建的房子。同时，性状只是帮助建立蓝图（界面）所规划的房屋的一种方式。

Interfaces

那么，我们为什么要使用接口呢？非常简单，接口使我们的代码更脆弱。如果您怀疑这个说法，可以问任何被迫维护那些不是针对接口编写的遗留代码的人。

接口是程序员和他/她的代码之间的契约。接口说：“只要你按我的规则玩，你可以实现我，不管你喜欢什么，我保证我不会破坏你的其他代码。”

作为一个例子，考虑一个真实世界的场景（没有汽车或小工具）：

You want to implement a caching system for a web application to cut down on
server load

首先写一个类来缓存使用APC的请求响应：

class ApcCacher
{
  public function fetch($key) {
    return apc_fetch($key);
  }
  public function store($key, $data) {
    return apc_store($key, $data);
  }
  public function delete($key) {
    return apc_delete($key);
  }
}

然后，在你的http响应对象中，在做所有的工作来产生实际的响应之前，检查缓存命中：

class Controller
{
  protected $req;
  protected $resp;
  protected $cacher;

  public function __construct(Request $req, Response $resp, ApcCacher $cacher=NULL) {
    $this->req    = $req;
    $this->resp   = $resp;
    $this->cacher = $cacher;

    $this->buildResponse();
  }

  public function buildResponse() {
    if (NULL !== $this->cacher && $response = $this->cacher->fetch($this->req->uri()) {
      $this->resp = $response;
    } else {
      // build the response manually
    }
  }

  public function getResponse() {
    return $this->resp;
  }
}

这种方法效果很好。但也许几个星期后，你决定使用基于文件的缓存系统而不是APC。现在，您必须更改控制器代码，因为您已经将控制器编程为使用ApcCacher类的功能，而不是用于表示ApcCacher功能的接口类。让我们说，而不是上面你已经使Controller类依赖于一个CacherInterface，而不是像这样具体的ApcCacher：

// your controller's constructor using the interface as a dependency
public function __construct(Request $req, Response $resp, CacherInterface $cacher=NULL)

为了配合你的定义，像这样定义：

interface CacherInterface
{
  public function fetch($key);
  public function store($key, $data);
  public function delete($key);
}

反过来，您同时拥有您的ApcCacher和您的新的FileCacher类实现CacherInterface，并且您编写Controller类来使用接口所需的功能。

这个例子（希望）展示了如何编程到一个接口允许你改变你的类的内部实现，而不必担心这些改变是否会破坏你的其他代码。

Traits

另一方面，性状只是一种重用代码的方法。接口不应该被认为是性格的互斥选择。实际上，创建满足界面所需功能的特性是理想的用例_ **。

当多个类共享相同的功能（可能由相同的接口指定）时，应该只使用特征。使用trait为单个类提供功能是没有意义的：只会混淆类的功能，而更好的设计会将特征的功能转移到相关的类中。

考虑下面的特质实现：

interface Person
{
    public function greet();
    public function eat($food);
}

trait EatingTrait
{
    public function eat($food)
    {
        $this->putInMouth($food);
    }

    private function putInMouth($food)
    {
        // digest delicious food
    }
}

class NicePerson implements Person
{
    use EatingTrait;

    public function greet()
    {
        echo 'Good day, good sir!';
    }
}

class MeanPerson implements Person
{
    use EatingTrait;

    public function greet()
    {
        echo 'Your mother was a hamster!';
    }
}

更具体的例子：想象接口讨论中的FileCacher和ApcCacher都使用相同的方法来确定缓存条目是否过时并应该被删除显然现实生活中并非如此，而是与之相适应）。你可以编写一个特性，并允许这两个类使用它来满足通用的接口要求。

最后要提醒的一点是：小心不要让性格过度。当独特的类实现就足够了的时候，特性通常被用作糟糕设计的拐杖。您应该限制特性来满足最佳代码设计的接口要求。
