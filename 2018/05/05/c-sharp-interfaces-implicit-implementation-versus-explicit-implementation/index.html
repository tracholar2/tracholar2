C＃接口。隐式实现与显式实现

By zhizi

Published 2018-05-05

Contents

在C＃中实现 隐式 和 显式 实现接口有什么区别？

什么时候应该使用隐式的，什么时候应该使用explicit？

有没有优点和/或缺点吗？

微软官方指南（从第一版框架设计指南）指出，
不建议使用显式实现 ，因为它给代码带来了意想不到的行为。

我认为这个指导方针在IoC之前的时期是非常有效的，当你不把它作为接口传递的时候。

任何人都可以触及这方面？

隐式是在通过类中的成员定义接口时使用的。 Explicit**
是在接口上的类中定义方法的时候。我知道这听起来令人困惑，但这里是我的意思：IList.CopyTo将隐式实现为：

public void CopyTo(Array array, int index)
{
    throw new NotImplementedException();
}

并明确地表示为：

void ICollection.CopyTo(Array array, int index)
{
    throw new NotImplementedException();
}

不同之处在于隐式地可以通过您的类来创建，当它被转换为该类时以及转换为接口时。显式实现允许它只能在作为接口本身进行转换时才可访问。

MyClass myClass = new MyClass(); // Declared as concrete class
myclass.CopyTo //invalid with explicit
((IList)myClass).CopyTo //valid with explicit.

我主要使用显式来保持实现干净，或者当我需要两个实现。但是不管我很少使用它。

我相信有更多的理由使用它/不使用它，别人会发布。

请参阅 下一个帖子 在这个线程中，每个背后都有很好的推理。

隐式定义只是将接口所要求的方法/属性等直接作为公共方法添加到类中。

显式定义强制只有当你直接使用接口而不是底层的实现时才暴露成员。这在大多数情况下是首选。

  • 通过直接使用界面，您不承认， 并将您的代码耦合到底层实现。
  • 如果您已经拥有一个公共属性名称， 你的代码，你想实现一个接口，也有一个 名称属性，明确地做它将保持两个分开。甚至 如果他们正在做同样的事情，我仍然委托明确 调用Name属性。你永远不知道，你可能想要改变 Name如何为普通类工作，以及如何使用Name，接口 财产稍后工作。
  • 如果你隐式地实现了一个接口，那么你的类现在就暴露了 新的行为可能只与客户有关 界面，这意味着你不能保持你的课程简洁 足够（我的意见）。
