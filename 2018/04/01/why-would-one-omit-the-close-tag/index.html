我一直在阅读，在文件末尾使用PHP关闭标签？>＆gt;的做法很差。在以下的上下文中头标题似乎是不相关的（这是目前唯一的一个很好的论点）：

Modern versions of PHP set the output_buffering flag in php.ini If output
buffering is enabled, you can set HTTP headers and cookies after outputting
html because returned code is not sent to the browser immediately.

每一本好的练习本和维基都以这个“规则”开始，但没有人提供很好的理由。 是否有另一个不错的理由来跳过结尾的php标记？

虽然我不记得任何其他原因，但比正常情况下发送标题可能会产生深远的影响。下面是刚才我想到的其中几个：

  • 当前的PHP版本可能会输出缓冲，而实际的 生产服务器 ，您将部署代码远比任何开发或测试机器更重要。他们并不总是倾向于立即遵循最新的PHP趋势。

  • 您可能会对难以理解的 功能损失 感到头疼。假设您正在实施某种付款网关，并在付款处理器成功确认后将用户重定向到特定的网址。如果出现某种PHP错误，甚至是警告，或者发生了额外的行结束，那么付款可能仍然是未处理的，并且用户可能看起来仍然没有收费。这也是不必要的重定向是邪恶的原因之一，如果要使用重定向，必须谨慎使用。

  • 即使在最新的版本中，您可能会在Internet Explorer中收到“页面加载取消”类型的错误。这是因为 AJAX 响应/ json包含了一些不应该包含的内容，因为在某些PHP文件中有多余的行结尾，正如我前几天遇到的一样。

  • 如果您的应用中有 文件下载 ，则可能会因此而中断。而且即使经过了几年，你也不会注意到，因为下载的具体破坏习惯取决于服务器，浏览器，文件的类型和内容（还有其他一些我不想让你感到厌烦的因素）

奖金：与这两个字符相关的几个陷阱（实际上是一个）：

  • 甚至一些知名的库可能在？>＆gt;之后包含多余的行结尾。一个例子是Smarty，即使是最新版本的2. 和3. 分支都有这个。因此，一如既往， 注意第三方代码 。奖金奖金：用于删除不必要的PHP结尾的正则表达式：在包含PHP代码的所有文件中，将（\ s * \？＆gt; \ s *）$替换为空文本。

你应该离开php结束标记（？>＆gt;）的原因是程序员不会意外地发送额外的换行字符。

你不应该离开php结束标记的原因是因为它会导致php标记的不平衡，任何半心的程序员都可以记住不要添加额外的空格。

所以对于你的问题：

Is there another good reason to skip the ending php tag?

不，没有另外一个很好的理由来跳过结尾的php标签。
