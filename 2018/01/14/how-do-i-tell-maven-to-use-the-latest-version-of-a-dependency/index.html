我如何告诉Maven使用最新版本的依赖关系？

By zhizi

Published 2018-01-14

Contents

在Maven中，依赖通常是这样设置的：

<dependency>
  <groupId>wonderful-inc</groupId>
  <artifactId>dream-library</artifactId>
  <version>1.2.3</version>
</dependency>

现在，如果您正在使用频繁发布的库，请不断更新＆lt; version＆gt;标签可能有点烦人。有什么办法告诉Maven总是使用最新的可用版本（从版本库）？

_这个答案只适用于Maven 2！上面提到的LATEST和RELEASE metaversions [。_

如果您始终想要使用最新版本，Maven有两个关键字可以用来替代版本范围。您应该谨慎使用这些选项，因为您不再控制所使用的插件/依赖项。

When you depend on a plugin or a dependency, you can use the a version value
of LATEST or RELEASE. LATEST refers to the latest released or snapshot version
of a particular artifact, the most recently deployed artifact in a particular
repository. RELEASE refers to the last non-snapshot release in the repository.
In general, it is not a best practice to design software which depends on a
non-specific version of an artifact. If you are developing software, you might
want to use RELEASE or LATEST as a convenience so that you don’t have to
update version numbers when a new release of a third-party library is
released. When you release software, you should always make sure that your
project depends on specific versions to reduce the chances of your build or
your project being affected by a software release not under your control. Use
LATEST and RELEASE with caution, if at all.

请参阅 Maven书籍的POM语法部分了解更多详情。或者在依赖版本范围上查看此文档，其中：

  • 方括号（[＆amp; ]）表示“closed”（包含）。
  • 圆括号（（＆amp; ））的意思是“open”（独占）。

这是一个例子说明各种选项。在Maven仓库中，com.foo：my-foo具有以下元数据：

<?xml version="1.0" encoding="UTF-8"?><metadata>
  <groupId>com.foo</groupId>
  <artifactId>my-foo</artifactId>
  <version>2.0.0</version>
  <versioning>
    <release>1.1.1</release>
    <versions>
      <version>1.0</version>
      <version>1.0.1</version>
      <version>1.1</version>
      <version>1.1.1</version>
      <version>2.0.0</version>
    </versions>
    <lastUpdated>20090722140000</lastUpdated>
  </versioning>
</metadata>

如果需要依赖该工件，您可以使用以下选项（其他 version ranges，这里只显示相关的）：

声明一个确切的版本（总是会解析为1.0.1）：

<version>[1.0.1]</version>

声明一个明确的版本（除非发生冲突，否则Maven将选择一个匹配版本，否则将永远解析为1.0.1）：

<version>1.0.1</version>

声明所有1.x的版本范围（目前将解析为1.1.1）：

<version>[1.0.0,2.0.0)</version>

声明开放式版本范围（将解析为2.0.0）：

<version>[1.0.0,)</version>

声明版本为LATEST（将解析为2.0.0）（从maven 3.x中移除）

<version>LATEST</version>

声明版本为RELEASE（将解析为1.1.1）（从maven 3.x中删除）：

<version>RELEASE</version>

请注意，默认情况下，您自己的部署将更新Maven元数据中的“latest”条目，但要更新“release”条目，您需要从[。你可以用“-Prelease-
profile”或者“-DperformRelease = true”做这个。

值得强调的是，任何允许Maven选择依赖版本（LATEST，RELEASE和版本范围）的方法都可能会让您打开时间问题，因为后来的版本可能会有不同的行为（例如，依赖项插件先前将默认值从true切换到false，结果令人困惑）。

因此，在发行版中定义精确版本通常是一个好主意。正如 Tim’s答案指出， maven-versions-plugin是一个方便的工具更新依赖版本，特别是版本：use-
latest-版本和版本：使用最新版本目标。

现在我知道这个话题已经老了，但是读到这个问题，OP提供了答案，看起来好像是 Maven Versions Plugin可能实际上是对他的问题的更好的回答：

特别是下面的目标可以使用：

  • 版本：use-latest-versions 在所有版本中搜索pom 这是一个更新的版本和 用最新的替换它们 版本
  • 版本：use-latest-releases 在pom中搜索所有非SNAPSHOT 版本已经更新了 释放并用它们替换它们 最新版本。
  • 版本：更新属性 更新a中定义的属性 项目，以便他们对应 最新的可用版本 特定的依赖关系。这可以 如果有一套依赖关系很有用 必须全部锁定到一个版本。

还提供了以下其他目标：

  • 版本：display-dependency-updates 扫描项目的依赖关系 产生这些报告 依赖关系更新 版本可用。
  • 版本：display-plugin-updates 扫描项目的插件和 产生这些插件的报告 有更新的版本可用。
  • 版本：update-parent 更新项目的父项部分 它引用了最新的 可用的版本。例如，如果 你用一个公司的根POM，这个 目标可以是有用的，如果你需要 确保你使用最新的 版本的公司根POM。
  • 版本：update-child-modules 更新 一个项目的子模块等等 版本匹配的版本 当前的项目。例如，如果你 有一个聚合pom也是 它的项目的父母 总量和儿童和 父母版本不同步，这 mojo可以帮助修复的版本 子模块。 （注意你可能需要 在-N选项中调用Maven 为了运行这个目标，如果你的 项目被打破得如此糟糕 由于版本不能建立 错配）。
  • 版本：锁定快照 在pom中搜索所有的“-NASHOT” 版本并用它们替换它们 当前的时间戳版本 -SNAPSHOT，例如-20090327.172306-4
  • 版本：unlock-snapshots 会搜索所有时间戳 锁定的快照版本和替换 他们与-SNAPSHOT。
  • 版本：resolve-ranges 使用版本范围和 解决了具体的范围 正在使用的版本。
  • 版本：使用版本 在pom中搜索所有的“-NASHOT”版本 已经发布和取代 他们与相应的释放 版本
  • 版本：use-next-releases 在pom中搜索所有非SNAPSHOT 版本已经更新了 释放并用它们替换它们 下一个版本。
  • 版本：use-next-versions 在pom中搜索所有版本 这是一个更新的版本和 将其替换为下一个版本。
  • 版本：提交 删除pom.xml.versionsBackup文件。形式 一半的内置“穷人” SCM”。
  • 版本：恢复 可以从中恢复pom.xml文件 pom.xml.versionsBackup文件。形式 一半的内置“穷人” SCM”。
