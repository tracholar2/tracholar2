如果解释Python，什么是.pyc文件？

By zhizi

Published 2018-01-14

Contents

我已经了解到Python是一种解释型语言…然而，当我查看我的Python源代码时，我会看到.pyc文件，Windows将其标识为“编译的Python文件”。这些来自哪里？

它们包含字节码，这是Python解释器编译源代码的地方。这个代码然后由Python的虚拟机执行。

Python的文档解释了这样的定义：

Python is an interpreted language, as opposed to a compiled one, though the
distinction can be blurry because of the presence of the bytecode compiler.
This means that source files can be run directly without explicitly creating
an executable which is then run.

I’ve been given to understand that Python is an interpreted language…

这种流行的模因是不正确的，或者说是建立在对（自然）语言水平的误解之上：类似的错误是说“圣经是一本精装书”。让我解释一下这个比喻…

“圣经”是指作为（实际的，物理的物体被确定为）书的一类的“一本书”。被认为是“圣经副本”的书籍应该有一些共同的基本内容（虽然即使是那些可以用不同的语言，不同的可接受的翻译，脚注的水平和其他注释）

  • 然而，这些书是完全可以在很多方面有所不同，这些方面不是被认为是基本的 -
    有约束力的，有约束力的颜色，打印中使用的字体，插图（如果有的话），宽可写边距或不，内置书签的数量和种类等等，等等。

“圣经”典型的_印刷很可能确实是精装书 -
毕竟，这是一本通常要反复阅读的书，在几个地方加了书签，通过查找给定的章节和指针等，一个好的精装书可以使一个给定的副本在这样的使用下持续更长的时间。然而，这些都是世俗的（实际的）问题，不能用来确定给定的实际书目对象是否是圣经的副本：平装印刷是完全可能的！

同样，Python在定义一类 实现 _
类的意义上是“一种语言”，它们在某些基本方面必须相似（语法，大多数语义除外那些明确允许它们不同的部分），但是完全允许在每个“实现”细节上有所不同 -
包括它们如何处理它们给出的源文件，是否将源编译成一些较低（如果是的话，是哪种形式，以及它们是否将这些编译后的表格保存到磁盘或其他地方），它们是如何执行所述表格的，等等。

传统的实现CPython通常简称为“Python”，但它只是几个产品质量实现中的一个，与Microsoft的IronPython（编译为CLR代码，即“.NET”
），Jython（编译为JVM代码），PyPy（用Python自己编写，可以编译成各种各样的“后端”形式，包括“即时生成”机器语言）。他们都是Python（==“Python语言的实现”），就像许多表面上不同的书对象都可以是圣经（==“圣经”的副本）。

如果您对CPython有兴趣，可以将源文件编译为特定于Python的低级表单（称为“字节码”），在需要时自动执行（当没有与源对应的字节码文件文件或字节码文件比源文件早，或者由不同的Python版本编译），通常会将字节码文件保存到磁盘（以避免将来重新编译它们）。
OTOH IronPython通常会编译为CLR代码（将它们保存到磁盘或不保存），Jython到JVM代码（将它们保存到磁盘或不保存） -
如果它保存，它将使用.class扩展名它们）。

然后，这些较低级别的表单随后被合适的“虚拟机”（也称为“解释器”）执行 - CPython VM，.Net运行时，Java
VM（又名JVM），如果适用的话。

因此，在这个意义上（典型的实现是做什么的），当且仅当C＃和Java是Python时，Python才是“解释型语言”：它们都有一个典型的实现策略，首先生成字节码，然后通过VM
/解释器。

更有可能的重点是汇编过程如何“沉重”，“慢”和“高”。 CPython被设计为尽可能快地尽可能轻地编译，尽可能少的仪式 -
编译器只做很少的错误检查和优化，所以它可以快速运行并且在少量内存中运行在需要时自动且透明地运行，而用户甚至不需要意识到大部分时间都在进行编辑。
Java和C＃在编译过程中通常会接受更多的工作（因此不会执行自动编译），以便更彻底地检查错误并执行更多的优化。它是一个灰度级的连续体，不是黑色或白色的情况，在某个给定的水平上设置一个阈值并且说只有在这个水平以上才称之为“汇编”是完全武断的
