避免！=空语句

By zhizi

发表于 2018-01-14

文章目录

在Java中编程时，我使用最多的惯例是在使用它之前测试object！= null。这是为了避免 NullPointerException。我发现代码非常难看，而且变得不可读。

有这个好的选择吗？

我想解决测试每个对象的必要性，如果你想访问这个对象的字段或方法。

例如：

if (someobject != null) {
    someobject.doCalc();
}

在这种情况下，我将避免NullPointerException，并且我不知道对象是否是null。这些测试出现在我的代码中，作为结果。

对我而言，这听起来像是一个相当普遍的问题，初级到中级开发人员在某些时候往往会面临这样的问题：他们要么不知道，要么不信任他们正在参与的合同，并且防御性地检查空值。另外，在编写他们自己的代码时，他们倾向于依靠返回空值来指示某事，因此要求调用者检查空值。

换句话说，有两个实例出现空检查：

  • 如果null是合同方面的有效答复，以及

  • 哪里不是有效的答复。

（2）很容易。使用assert语句（断言）或允许失败（例如 NullPointerException）。断言是1.4中添加的一个使用率很低的Java特性。语法是：

assert <condition>

或

assert <condition> : <object>

＆lt; condition＆gt;是一个布尔表达式，＆lt; code＆gt;＆lt; object＆gt; 是一个将toString（）`在错误。

如果条件不成立，assert语句会抛出Error（）。默认情况下，Java会忽略断言。您可以通过将选项-ea传递给JVM来启用断言。您可以启用和禁用个别类和包的声明。这意味着您可以在开发和测试的同时使用断言验证代码，并在生产环境中禁用它们，虽然我的测试显示接近断言的性能影响。

在这种情况下不使用断言是可以的，因为代码将会失败，如果使用断言，会发生什么情况。唯一的区别是，断言可能会更快地发生，并且可能会有更多的信息，这可能会帮助你弄清楚为什么它会发生，如果你不期望它。

（1）有点困难。如果你无法控制你打电话的代码，那么你就被卡住了。如果null是有效的响应，则必须检查它。

如果是你所控制的代码，那么情况往往是这样的，那么情况就不一样了。避免使用空值作为回应。使用返回集合的方法，很简单：几乎总是返回空集合（或数组）而不是null。

非集合可能会更困难。考虑这个例子：如果你有这些接口：

public interface Action {
  void doSomething();
}

public interface Parser {
  Action findAction(String userInput);
}

在这里，解析器需要原始的用户输入，并找到要做的事情，也许如果你正在实现一个命令行界面的东西。现在，如果没有适当的操作，您可以使合同返回null。这导致你正在谈论的空检查。

另一种解决方案是永远不会返回null，而是使用空对象模式：
p>

public class MyParser implements Parser {
  private static Action DO_NOTHING = new Action() {
    public void doSomething() { /* do nothing */ }
  };

  public Action findAction(String userInput) {
    // ...
    if ( /* we can't find any actions */ ) {
      return DO_NOTHING;
    }
  }
}

比较

Parser parser = ParserFactory.getParser();
if (parser == null) {
  // now what?
  // this would be an example of where null isn't (or shouldn't be) a valid response
}
Action action = parser.findAction(someInput);
if (action == null) {
  // do nothing
} else {
  action.doSomething();
}

到

ParserFactory.getParser().findAction(someInput).doSomething();

这是一个更好的设计，因为它导致更简洁的代码。

也就是说，对于findAction（）方法来说，抛出一个有意义的错误信息的Exception是完全合适的 - 特别是在这种情况下，你依赖于用户输入。
findAction方法抛出一个Exception比调用方法抛出一个简单的NullPointerException没有任何解释更好。

try {
    ParserFactory.getParser().findAction(someInput).doSomething();
} catch(ActionNotFoundException anfe) {
    userConsole.err(anfe.getMessage());
}

或者如果你认为try / catch机制太丑陋了，而不是Do Nothing，那么你的默认行为应该给用户提供反馈。

public Action findAction(final String userInput) {
    /* Code to return requested Action if found */
    return new Action() {
        public void doSomething() {
            userConsole.err("Action not found: " + userInput);
        }
    }
}

如果您使用（或计划使用）Java IDE，请 JetBrains IntelliJ IDEA使用他们开发的一些特定的注释。

基本上，你有@Nullable和@NotNull。

您可以在方法和参数中使用，如下所示：

@NotNull public static String helloWorld() {
    return "Hello World";
}

或

@Nullable public static String helloWorld() {
    return "Hello World";
}

第二个例子不能编译（在IntelliJ IDEA中）。

在另一段代码中使用第一个hello World（）函数：

public static void main(String[] args)
{
    String result = helloWorld();
    if(result != null) {
        System.out.println(result);
    }
}

现在IntelliJ IDEA编译器会告诉你这个检查是无用的，因为helloWorld（）函数不会返回null
使用参数

void someMethod(@NotNull someParameter) { }

如果你写了类似：

someMethod(null);

这不会编译。

最后一个使用@Nullable
的例子

@Nullable iWantToDestroyEverything() { return null; }

这样做

iWantToDestroyEverything().something();

你可以肯定，这不会发生。 ：）

让编译器检查一些比平时更多的东西，并强制你的合约变得更强大，这是一个不错的方法。不幸的是，这并不是所有编译器都支持的。

在IntelliJ IDEA 10.5及以上版本中，他们添加了对任何其他@Nullable @NotNull实现的支持。

@NotNull注释 。
