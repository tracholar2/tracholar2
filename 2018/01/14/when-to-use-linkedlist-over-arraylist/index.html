何时通过ArrayList使用LinkedList？

By zhizi

Published 2018-01-14

Contents

我一直只使用一个：

List<String> names = new ArrayList<String>();

我使用接口作为 portability 的类型名称，所以当我提出这样的问题时，我可以重写我的代码。

什么时候应该使用链接列表来覆盖ArrayList，反之亦然？

在ArrayList中使用ArrayList比LinkedList更适合_ 。不确定 - 只需从 ArrayList
`开始。

LinkedList 和 ArrayList 是List接口的两个不同的实现。 LinkedList 用一个双向链表实现它。ArrayList `用一个动态调整大小的数组来实现它。

与标准的链表和数组操作一样，各种方法也会有不同的算法运行时间。

对于 ` LinkedList＆lt; E＆gt;

get（int index）`是 O（n / 4） 平均

  • add（E element）是 O（1）

  • add（int index，E element）是 O（n / 4） average
    但是index = 0时的 O（1） <-– LinkedList

  • remove（int index）是 O（n / 4） 平均值

  • Iterator.remove（）是 O（1） <-– LinkedList
    ListIterator.add（E element）是 O（1） <-– LinkedList 的主要优点`
    注意：O（n / 4）_是平均值，O（1）是最好的情况（例如index = 0），O（n / 2）最差情况（列表中）

对于 ` ArrayList＆lt; E＆gt;

ArrayList的主要优点

因为数组必须调整大小，因此add（E元素）`是O（1），但是O（n）是最坏的情况。并复制

  • add（int index，E element）是 O（n / 2） 平均值
  • remove（int index）是 O（n / 2） 平均值
  • Iterator.remove（）是 O（n / 2） 平均
  • ListIterator.add（E元素）是 O（n / 2） 平均值

O（n / 2）是平均值，O（1）是最好的情况（列表的结尾），O（n） 最坏的情况（列表开始）

LinkedList＆lt; E＆gt;
允许使用迭代器_进行常量插入或删除，但只能按顺序访问元素。换句话说，您可以向前或向后移动列表，但在列表中查找位置需要与列表大小成比例的时间。 Javadoc说：“索引到列表中的操作将从开始或结束遍历列表中，无论哪个更接近”_，因此这些方法是 _O（n / 4）_ 平均，虽然 index = 0
`的O（1）_。

另一方面，ArrayList＆lt; E＆gt;
允许快速的随机读取访问，所以你可以在任何时间获取元素。但是，除了最终目标之外，任何地方都可以加入或者删除，这要求将所有后面的要素都转移，要么开放，要么填补空白。此外，如果添加的元素数量多于底层数组的容量，则会分配一个新数组（1.5倍大小），并将旧数组复制到新数组中，从而添加到ArrayList `在最坏的情况下是O（n）_，但平均值是不变的。

所以根据你打算做的操作，你应该选择相应的实现。迭代任何一种List实际上同样便宜。 （对ArrayList进行迭代在技术上比较快，但除非您对性能敏感，否则不应该担心 - 它们都是常量。）

使用LinkedList的主要好处是在重用现有的迭代器来插入和删除元素时产生。这些操作可以在 O（1）
中完成，只需在本地更改列表。在数组列表中，数组的其余部分需要被 移动 （即复制）。另一方面，在LinkedList中寻找意味着在最坏情况下遵循
O（n / 2） 中的链接，而在ArrayList期望的位置可以通过数学计算并在 O（1） 。

中访问

使用LinkedList的另一个好处是，当你从列表的头部添加或者删除的时候会产生一个好处，因为这些操作是 O（1） ，而 > O（n）
用于ArrayList。请注意，ArrayDeque可能是LinkedList的一个很好的替代方案，用于添加和删除头部，但不是List。

另外，如果你有大的列表，请记住内存使用情况也是不同的。 LinkedList的每个元素都有更多的开销，因为指向下一个元素和前一个元素的指针也被存储。
ArrayLists没有这个开销。然而，无论元素是否被添加，ArrayLists占用了为容量分配的内存量。

ArrayList的默认初始容量非常小（从Java 1.4 -
1.8中的10个）。但是由于底层实现是一个数组，所以如果添加了很多元素，则必须调整数组的大小。为了避免在知道要添加大量元素时调整大小的成本，请使用更高的初始容量构建ArrayList。

到目前为止，除了一般认为LinkedList比多得多的一般共识外，没有人似乎已经解决了每个列表的内存占用情况，所以我做了一些数字计算，以确切地证明两个列表占用了多少N空引用。

由于引用在其相关系统上是32位或64位（即使为空），我已经为32位和64位LinkedLists和ArrayLists
注意：ArrayList行显示的大小是 trimmed lists - 实际上，支持数组的容量在 < code> ArrayList
`通常大于当前的元素数。

注意2：（感谢BeeOnRope）_由于CompressedOops现在是从JDK6以上的版本开始默认的，64位机器下面的值基本上与它们的32位同行，除非你特别关掉它。

结果清楚地表明，LinkedList比ArrayList 要多得多，特别是元素数量非常高。如果内存是一个因素，请避开 LinkedLists
`。

我使用的公式如下，让我知道如果我做错了什么，我会修好它。对于32或64位系统，’b’是4或8，’n’是元素的数量。注意mods的原因是因为java中的所有对象都占用8个字节的倍数，不管是否全部使用。

<代码>的ArrayList ：

ArrayList object header + size integer + modCount integer + array reference + (array oject header + b * n) + MOD(array oject, 8) + MOD(ArrayList object, 8) == 8 + 4 + 4 + b + (12 + b * n) + MOD(12 + b * n, 8) + MOD(8 + 4 + 4 + b + (12 + b * n) + MOD(12 + b * n, 8), 8)

链接列表：

LinkedList object header + size integer + modCount integer + reference to header + reference to footer + (node object overhead + reference to previous element + reference to next element + reference to element) * n) + MOD(node object, 8) * n + MOD(LinkedList object, 8) == 8 + 4 + 4 + 2 * b + (8 + 3 * b) * n + MOD(8 + 3 * b, 8) * n + MOD(8 + 4 + 4 + 2 * b + (8 + 3 * b) * n + MOD(8 + 3 * b, 8) * n, 8)
