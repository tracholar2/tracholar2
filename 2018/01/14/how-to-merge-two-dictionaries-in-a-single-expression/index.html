我有两个Python字典，我想写一个表达式，返回这两个字典，合并。 update（）方法是我所需要的，如果它返回的结果，而不是就地修改字典。

>>> x = {'a':1, 'b': 2}
>>> y = {'b':10, 'c': 11}
>>> z = x.update(y)
>>> print(z)
None
>>> x
{'a': 1, 'b': 10, 'c': 11}

如何在z中获得最终的合并字典，而不是x？

（更加清楚的是，dict.update（）的最后一次冲突处理也是我正在寻找的。）

How can I merge two Python dictionaries in a single expression?

Explanation

假如你有两个词典，你想把它们合并成一个新的词典，而不改变原来的词典：

x = {'a': 1, 'b': 2}
y = {'b': 3, 'c': 4}

理想的结果是得到一个新的字典（z），其中的值被合并，第二个字典的值覆盖了第一个字典。

>>> z
{'a': 1, 'b': 3, 'c': 4}

一个新的语法，在 PEP 448 和自Python
3.5开始，是

z = {**x, **y}

这确实是一个单一的表达。现在显示为在 3.5版本发布计划中实施，PEP 478，现在已经进入 Python
3.5中的新功能文档。

然而，由于许多组织仍然在Python 2上，所以您可能希望以向后兼容的方式来完成此操作。 Python 2和Python
3.0-3.4中提供的经典Pythonic方法是将其分成两步：

z = x.copy()
z.update(y) # which returns None since it mutates z

在这两种方法中， y 将会是第二个，它的值会替换 x 的值，因此‘b’会指向<代码> 3在我们的最终结果。

Not yet on Python 3.5, but want a single expression

如果您还没有在Python 3.5中，或者需要编写向后兼容的代码，并且希望在单个表达式_中使用，那么正确的方法是将其放在一个函数中：

def merge_two_dicts(x, y):
    """Given two dicts, merge them into a new dict as a shallow copy."""
    z = x.copy()
    z.update(y)
    return z

然后你有一个表达式：

z = merge_two_dicts(x, y)

你也可以创建一个函数来合并一个从零到非常大的数字的未定义的字符串：

def merge_dicts(*dict_args):
    """
    Given any number of dicts, shallow copy and merge into a new dict,
    precedence goes to key value pairs in latter dicts.
    """
    result = {}
    for dictionary in dict_args:
        result.update(dictionary)
    return result

这个函数可以在Python 2和3中用于所有的字典。例如给定字典a为g：

z = merge_dicts(a, b, c, d, e, f, g)

g中的键值对将优先于dicts a转换为f，等等。

Critiques of Other Answers

不要使用您在以前接受的答案中看到的内容：

z = dict(x.items() + y.items())

在Python 2中，为每个字典在内存中创建两个列表，在内存中创建第三个列表，长度等于前两个列表的长度，然后放弃所有三个列表来创建字典。 在Python
3中，这会失败 ，因为您要添加两个dict_items对象，而不是两个列表

>>> c = dict(a.items() + b.items())
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: unsupported operand type(s) for +: 'dict_items' and 'dict_items'

你必须明确地将它们创建为列表，例如z = dict（list（x.items（））+ list（y.items（）））。这是资源和计算能力的浪费。

同样，Python 2.7中的items（）中的items（）的联合也会在值不可更改的对象（如列表，例）。即使你的值是可散列的，
因为集合在语义上是无序的，所以关于优先级的行为是未定义的。所以不要这么做：

>>> c = dict(a.items() | b.items())

这个例子演示了值不可用时会发生什么：

>>> x = {'a': []}
>>> y = {'b': []}
>>> dict(x.items() | y.items())
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: unhashable type: 'list'

下面是一个例子，其中y应该优先，但是由于集合的任意顺序，x的值被保留：

>>> x = {'a': 2}
>>> y = {'a': 1}
>>> dict(x.items() | y.items())
{'a': 2}

另一个你不应该使用的黑客：

z = dict(x, **y)

这使用了dict构造函数，并且非常快速和高效地存储内存（甚至比我们的两步过程稍微多一些），但除非您确切地知道这里发生了什么（也就是第二个字典是作为关键字参数传递给字典构造函数的），它很难阅读，这不是预期的用法，所以它不是Pythonic。

以下是使用方法的一个例子，在django中修正[ <一>

Dicts的目的是取得可哈希键（例如frozensets或tuples），但是 当键不是字符串时，这个方法在Python 3中失败。

>>> c = dict(a, **b)
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: keyword arguments must be strings

从邮寄名单，Guido van Rossum ，这个语言的创造者写道：

I am fine with declaring dict({}, {1:3}) illegal, since after all it is
abuse of the mechanism.

和

Apparently dict(x, **y) is going around as “cool hack” for “call x.update(y)
and return x”. Personally I find it more despicable than cool.

这是我的理解（以及对[ （** y）`的用途是为了可读性目的创建字典，例如：

dict(a=1, b=10, c=11)

而不是

{'a': 1, 'b': 10, 'c': 11}

Response to comments

Despite what Guido says, dict(x, **y) is in line with the dict
specification, which btw. works for both Python 2 and 3. The fact that this
only works for string keys is a direct consequence of how keyword parameters
work and not a short-comming of dict. Nor is using the operator in this
place an abuse of the mechanism, in fact was designed precisely to pass
dicts as keywords.

同样，当键不是字符串时，它不适用于3。隐式调用约定是命名空间采用普通的字典，而用户只能传递字符串的关键字参数。所有其他可调用的强制执行它。 dict在Python 2中打破了这种一致性：

>>> foo(**{('a', 'b'): None})
Traceback (most recent call last):
  File "<stdin>", line 1, in <module>
TypeError: foo() keywords must be strings
>>> dict(**{('a', 'b'): None})
{('a', 'b'): None}

这种不一致在Python的其他实现（Pypy，Jython，IronPython）上是不好的。因此它在Python
3中被修复，因为这个用法可能是一个突破性的改变。

我向你提出，故意编写只能在一种语言版本中工作的代码，或者只能在某些特定的约束条件下工作的恶意代码才是恶意的。

另一个评论：

dict(x.items() + y.items()) is still the most readable solution for Python

  1. Readability counts.

我的回应：如果我们真的关心可读性，merge_two_dicts（x，y）实际上对我来说似乎更加清晰。它不是向前兼容的，因为Python
2越来越被弃用。

Less Performant But Correct Ad-hocs

这些方法性能较差，但会提供正确的行为。 它们会比copy和update性能低得多，或者是新的拆包，因为它们在更高的抽象层次上迭代每个键值对，但是他们尊重优先顺序（后面的字典有优先权）

你也可以在dict理解的范围内手工链接dicts：

{k: v for d in dicts for k, v in d.items()} # iteritems in Python 2.7

或者在Python 2.6（也许早在引入生成器表达式的时候就是2.4）：

dict((k, v) for d in dicts for k, v in d.items())

itertools.chain `会按照正确的顺序将迭代器链接到键值对上：

import itertools
z = dict(itertools.chain(x.iteritems(), y.iteritems()))

Performance Analysis

我只会做已知行为正确的用法的性能分析。

import timeit

以下是在Ubuntu 14.04上完成的

在Python 2.7（系统Python）中：

>>> min(timeit.repeat(lambda: merge_two_dicts(x, y)))
0.5726828575134277
>>> min(timeit.repeat(lambda: {k: v for d in (x, y) for k, v in d.items()} ))
1.163769006729126
>>> min(timeit.repeat(lambda: dict(itertools.chain(x.iteritems(), y.iteritems()))))
1.1614501476287842
>>> min(timeit.repeat(lambda: dict((k, v) for d in (x, y) for k, v in d.items())))
2.2345519065856934

在Python 3.5（deadsnakes PPA）中：

>>> min(timeit.repeat(lambda: {**x, **y}))
0.4094954460160807
>>> min(timeit.repeat(lambda: merge_two_dicts(x, y)))
0.7881555100320838
>>> min(timeit.repeat(lambda: {k: v for d in (x, y) for k, v in d.items()} ))
1.4525277839857154
>>> min(timeit.repeat(lambda: dict(itertools.chain(x.items(), y.items()))))
2.3143140770262107
>>> min(timeit.repeat(lambda: dict((k, v) for d in (x, y) for k, v in d.items())))
