列表过滤：列表理解与lambda +过滤器

By zhizi

Published 2018-01-14

Contents

我碰巧发现自己有一个基本的过滤需求：我有一个列表，我必须通过项目的属性过滤它。

我的代码看起来像这样：

my_list = [x for x in my_list if x.attribute == value]

然而，我想，这样写是不是更好？

my_list = filter(lambda x: x.attribute == value, my_list)

它更具可读性，如果性能需要，lambda可以被拿出来获得一些东西。

问题是：在使用第二种方法时是否有任何警告？任何性能差异？我是否完全错过了Pythonic Way
?,应该用另一种方式来做（比如使用itemgetter而不是lambda）？

对于不同的人来说，有多少美女是不一样的。我发现列表的理解比filter + lambda要清楚得多，但是使用哪一个更容易。但是，请停止提供已经用于内置插件的变量名称，这是令人困惑的。

有两件事可能会减慢你使用filter。

第一个是函数调用的开销：只要你使用Python函数（不管是由def还是lambda创建的），过滤器可能会比列表理解。这几乎肯定是不够的，你不应该考虑性能，直到你已经定时代码，发现它是一个瓶颈，但差异将在那里。

另一个可能适用的开销是lambda被强制访问一个作用域变量（value）。这比访问本地变量慢，在Python
2.x中，列表理解只能访问局部变量。如果您使用的是Python 3.x，则列表理解运行在一个单独的函数中，因此它也将通过闭包访问value，这种差异将不适用。

另一个选择是使用生成器而不是列表理解：

def filterbyvalue(seq, value):
   for el in seq:
       if el.attribute==value: yield el

然后在你的主代码中（这是可读性真正重要的地方），你用一个有希望的函数名替换了列表理解和过滤器。

这在Python中是一个有点宗教的问题。即使 Guido考虑删除map，filter和reduce from Python
3 ，有足够的反冲，最后只有reduce被从内建-ins functools.reduce。

我个人觉得列表理解更容易阅读。因为所有的行为都在表面而不是在过滤器函数内部，所以更明确的是从表达式[i for i in i in i.attribute == value]中发生了什么。

我不会太担心这两种方法之间的性能差异，因为它是微不足道的。如果证明是应用程序中的瓶颈，那么我真的只会优化它，这是不可能的。

此外，自 BDFL
要` filter **代码>从语言中消失，然后肯定会自动使列表解析更Pythonic; - ）
