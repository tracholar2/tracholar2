Redux的作者在这里！

Redux不是和Flux不同的 __ 。总的来说，它具有相同的体系结构，但是Redux能够通过使用Flux使用回调注册的功能组合来减少一些复杂的角落。

在Redux中没有什么根本的不同，但是我发现Redux使得某些抽象更易于实现，或者至少可以实现，这在Flux中很难或者不可能实现。

Reducer Composition

以分页为例。我的 Flux + React Router示例处理分页，但是代码非常糟糕。其糟糕的原因之一是Flux使得跨商店重用功能是不自然的。**如果两个商店需要处理分页以响应不同的操作，则它们需要从公共基本商店继承！当你使用继承的时候，你将自己锁定在一个特定的设计中），或者从处理函数中调用一个函数，这个函数需要以某种方式在Flux存储的私有状态下运行。整个事情是混乱的（尽管在可能的范围内）。

另一方面，使用Redux分页是自然的减速器组成。它是减速器，所以你可以写一个减速器工厂，生成分页减速器，然后在你的reducer树中使用它。关键是为什么它很容易是因为Flux中的
，商店是平坦的，但是在Redux中，reducer可以通过功能组合嵌套，就像React组件可以嵌套一样。

这种模式还可以实现诸如非用户代码撤销/重做的精彩功能。
你能想象将Undo / Redo插入到两行代码的Flux应用程序中吗？几乎不。有了Redux，它又是
，这要归功于reducer组合模式。我需要强调一下，没有什么新东西 - 这是[榆树它本身就受到了Flux的影响。

Server Rendering

人们已经在服务器上使用Flux进行渲染了，但是看到我们有20个Flux库，每个都试图让服务器渲染“越来越难”，或许Flux在服务器上有一些粗糙的边缘。事实是，Facebook没有做太多的服务器渲染，所以他们一直没有十分关心它，并依靠这个生态系统来使其更容易。

在传统的Flux中，商店是单身人士。这意味着很难将服务器上不同请求的数据分开。不是不可能的，但很难。这就是为什么大多数Flux库（以及新的 Flux
Utils ）现在建议你使用类而不是单例，所以你可以为每个请求实例化商店。

Flux还需要解决以下问题（无论是您自己还是您最喜欢的Flux库的帮助，例如 Flummox或 Alt）：

  • 如果商店是类，如何根据请求创建和销毁调度程序？我什么时候注册商店？
  • 我如何从商店获取数据并在客户端补充水分？我是否需要为此执行特殊的方法？

无可否认，Flux框架（不是vanilla Flux）可以解决这些问题，但是我发现它们过于复杂。例如， Flummox要求你实现serialize（）和在您的商店中反序列化（）。 Alt通过提供 takeSnapshot（）来解决这个问题，状态在JSON树中。

Redux更进一步：由于只有一个商店（由许多reducer管理），因此您不需要任何特殊的API来管理（再）水合。**您不需要到“冲洗”或“水电”商店，只有一个商店，您可以阅读其当前状态，或创建一个新的状态的新商店。每个请求获取一个单独的商店实例。
阅读更多关于使用Redux的服务器渲染。

再次，这是Flux和Redux中可能的情况，但Flux库通过引入大量API和约定来解决这个问题，Redux甚至不需要解决这个问题，因为它没有首先由于简单的概念问题。

Developer Experience

我其实并不打算让Redux成为一个流行的Flux库 - 我是在写我的[可以通过消除动作来改变减速器代码甚至是“改变过去”，并且看到状态正在被重新计算。**

我还没有看到能够做到这一点的单个Flux库。 React Hot Loader也不会让你这样做 -
事实上，如果你编辑Flux商店，它会中断，因为它不知道如何处理它们。

当Redux需要重新加载Reducer代码时，它会调用 replaceReducer（ ），应用程序将使用新代码运行。在Flux中，数据和函数被Flux商店纠缠在一起，所以你不能“去替换函数”。此外，你必须以某种方式重新注册与Dispatcher的新版本

  • Redux甚至没有。

Ecosystem

Redux有一个丰富且快速增长的生态系统。这是因为它提供了一些扩展点，例如中间件。它的设计与用例如日志记录，支持承诺，[ Observables <
a>，路由，不可变性开关检查，[持久性记住，等等。并不是所有这些都会变得有用，但是能够访问一系列可以轻松组合在一起工作的工具是非常好的。

Simplicity

Redux保留了Flux的所有优点（记录和重播动作，单向数据流，相关突变），并且在不引入Dispatcher和商店注册的情况下增加了新的优点（简单的撤销 -
重做，热重载）。

保持简单是很重要的，因为它让你保持理智，同时实现更高层次的抽象。

与大多数Flux库不同，Redux API表面很小。如果您删除了开发人员的警告，注释和完整性检查，则99行。没有棘手的异步代码来调试。

基本上，Flux有多个存储计算状态变化，以响应与组件的UI /
API交互，并将这些更改作为组件可以订阅的事件广播。在Redux中，每个组件只订购一个商店。海事组织至少像Redux一样，通过统一（或减少，如Redux所说的）将数据流回到组件来进一步简化和统一数据流
