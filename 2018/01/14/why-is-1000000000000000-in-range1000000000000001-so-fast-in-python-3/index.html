为什么在Python 3中“1000000000000000（1000000000000001）”这么快？

By zhizi

Published 2018-01-14

Contents

据我所知，range（）函数实际上是 Python
3中的一个对象类型，即时生成其内容，类似于生成器。

如果是这样的话，我会预料到下面这一行会花费过多的时间，因为为了确定是否在这个范围内，需要生成一个四十亿个值：

1000000000000000 in range(1000000000000001)

此外：似乎不管我添加了多少个零，计算或多或少需要相同的时间量（基本上是瞬时的）。

我也尝试过这样的事情，但是计算仍然几乎是即时的：

1000000000000000000000 in range(0,1000000000000000000001,10) # count by tens

如果我试图实现我自己的范围功能，结果是不是很好！

def my_crappy_range(N):
    i = 0
    while i < N:
        yield i
        i += 1
    return

range（）`是什么使得它如此之快？

编辑：事实证明，这是一个比我预想的更细致的话题 - 在range（）优化的背后似乎有点历史。

Martijn
Pieters的答案是因为它的完整性而被选中的，但也可以参考abarnert的第一个答案，以便很好地讨论在Python中range是一个完整的 序列 3，关于__的潜在不一致性的一些信息/警告包含Python实现中的__函数优化。 abarnert的其他答案更详细一些，并提供了有关Python
3优化背后的历史感兴趣的链接（缺乏Python 2中xrange的优化）。
通过戳和通过wim为有兴趣的人提供相关的C源代码和解释。

Python 3 range（）对象不会立即生成数字;它是一个智能序列对象，可以根据需要产生数字_
。所有它包含的是你的开始，停止和步骤值，然后当你迭代对象时，每次迭代计算下一个整数。

这个对象还实现了 object .__ contains __ hook和
计算 是否您的号码是其范围的一部分。计算是一个O（1）常数时间操作。从来没有必要扫描范围内的所有可能的整数。

从 range（）对象文档： p>

The advantage of the range type over a regular list or tuple is that a
range object will always take the same (small) amount of memory, no matter the
size of the range it represents (as it only stores the start, stop and
step values, calculating individual items and subranges as needed).

所以至少你的range（）对象会这样做：

class my_range(object):
    def __init__(self, start, stop=None, step=1):
        if stop is None:
            start, stop = 0, start
        self.start, self.stop, self.step = start, stop, step
        if step < 0:
            lo, hi = stop, start
        else:
            lo, hi = start, stop
        self.length = ((hi - lo - 1) // abs(step)) + 1

    def __iter__(self):
        current = self.start
        if self.step < 0:
            while current > self.stop:
                yield current
                current += self.step
        else:
            while current < self.stop:
                yield current
                current += self.step

    def __len__(self):
        return self.length

    def __getitem__(self, i):
        if i < 0:
            i += self.length
        if 0 <= i < self.length:
            return self.start + i * self.step
        raise IndexError('Index out of range: {}'.format(i))

    def __contains__(self, num):
        if self.step < 0:
            if not (self.stop < num <= self.start):
                return False
        else:
            if not (self.start <= num < self.stop):
                return False
        return (num - self.start) % self.step == 0

这仍然缺少一些真正的range（）支持的东西（如.index（）或.count（）方法，哈希，平等测试或切片），但应该给你一个想法。

我还简化了__ contains __实现，只关注整型测试;如果给一个真正的range（）对象一个非整数值（包括int的子类），就会启动慢速扫描来查看是否存在匹配如果对包含的所有值列表使用包含测试。这样做是为了继续支持其他刚好支持用整数进行等式测试的数字类型，但是也不希望支持整数算术。请参阅实施遏制测试的原始Python问题。

这里的根本的误解是认为range是一个生成器。不是。事实上，它不是任何一种迭代器。

你可以很简单地告诉你：

>>> a = range(5)
>>> print(list(a))
[0, 1, 2, 3, 4]
>>> print(list(a))
[0, 1, 2, 3, 4]

如果它是一个生成器，重复一次就会耗尽它：

>>> b = my_crappy_range(5)
>>> print(list(b))
[0, 1, 2, 3, 4]
>>> print(list(b))
[]

实际上，range是一个序列，就像一个列表。你甚至可以测试这个：

>>> import collections.abc
>>> isinstance(a, collections.abc.Sequence)
True

这意味着它必须遵循作为序列的所有规则：

>>> a[3]         # indexable
3
>>> len(a)       # sized
5
>>> 3 in a       # membership
True
>>> reversed(a)  # reversible
<range_iterator at 0x101cd2360>
>>> a.index(3)   # implements 'index'
3
>>> a.count(3)   # implements 'count'
1

range 与 list 之间的区别在于 range 是 _lazy_ 或 _动态序列;它不记得它的所有值，它只是记住它的 start
， stop 和 step `，并根据需要在<代码> 的GetItem

（注意，如果你 （iter（a）），你会注意到 range 使用了相同的 listiterator 类型为 list 。这是如何工作的？一个listiterator 没有使用任何关于 list 的特殊内容，除了它提供了一个C实现 getitem ，所以它对 range
`也能正常工作。）

现在，没有什么说Sequence .__ contains __必须是常量时间 - 事实上，对于像list这样的序列的明显例子，事实并非如此。但没有什么说它不能 _是。实现range .__包含__以便在数学上检查它（（val-start）％step，但处理负面步骤有一些额外的复杂性）比实际更容易生成和测试所有的值，那么为什么不应该这样做呢？

但是在这种语言中似乎没有任何东西可以保证这种情况发生。正如Ashwini
Chaudhari所指出的那样，如果你给它一个非整数值，而不是转换为整数并进行数学测试，那么它将回落到迭代所有值并逐一比较它们。只是因为CPython
3.2+和PyPy 3.x版本碰巧包含了这个优化，而且这是一个明显的好主意，并且容易做到，IronPython或者NewKickAssPython
3.x没有任何理由不能忽略它。 （实际上CPython 3.0-3.1 _没有包含它。）

如果range实际上是一个生成器，比如my_crappy_range，那么用这种方法测试__ contains __是没有意义的，或者至少它的道理是不明显的。如果你已经迭代了前三个值，发生器是1仍然是测试1是否会导致它迭代并消耗所有的值到1（或者达到第一个值＆gt; = 1）？
