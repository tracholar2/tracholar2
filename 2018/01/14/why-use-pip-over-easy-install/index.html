为什么使用易于安装的点子？ [关闭]

By zhizi

发表于 2018-01-14

文章目录

鸣叫：

Don’t use easy_install, unless you like stabbing yourself in the face. Use
pip.

为什么使用pip over easy_install？ PyPI和软件包作者大部分都不是？如果作者上传垃圾源tarball（例如：缺少文件，没有setup.py）到PyPI，那么pip和easy_install都将失败。除了表面上的差异之外，为什么Python人（比如上面的tweet）似乎强烈地赞成通过easy_install进行点击？

（假设我们正在讨论由社区维护的Distribute包中的easy_install）

这里的许多答案已经过时了（虽然[ Daniel Roseman最初接受的答案不是） 。以下是当前状态：

  • 二进制包现在不仅在PyPI上，而且在第三方存储库（如 .whl `文件） uci.edu/~gohlke/pythonlibs/“rel =”noreferrer“> Christoph Gohlke的Windows扩展包。 pip可以处理车轮; easy_install不能。
  • 虚拟环境（内置3.4，或者可以通过添加到2.6 + / 3.1 + virtualenv ）已经成为一个非常重要和突出的工具（并在官方文档）;它们包括pip，但是甚至不能用easy_install正常工作。
  • 不再维护包含easy_install的distribute包。它对setuptools的改进已经合并到setuptools中。尝试安装distribute将只安装setuptools。
  • easy_install本身只是准维护。
  • 所有pip过去不如easy_install - 从解压源代码树，从DVCS回购等安装的情况都是早就没了;你可以pip install。，pip install git + https：//。
  • pip附带了来自python.org的官方Python 2.7和3.4+包，默认情况下包含了一个pip bootstrap。 >
  • 关于安装，使用和构建软件包的各种不完整的文档已被[ Python打包用户指南一>。 Python自己的安装Python模块的文档现在遵循本用户指南，并明确地调用out pip作为“首选安装程序”。
  • 多年来，其他新特性已被添加到pip中，这些特性永远不会存在于easy_install中。例如，pip通过构建一个需求文件，然后在每一边安装一个命令，就可以很容易地克隆你的站点包。或者将您的需求文件转换为本地回购以用于内部开发。依此类推。

我知道在2015年使用easy_install的唯一好理由是使用苹果预装的Python版本OS X
10.5-10.8的特殊情况。自10.5以来，苹果已经包含了easy_install，但从10.10开始，它们仍然不包含pip。对于10.9以上版本，您仍然应该使用get-pip.py，但是对于10.5-10.8，这存在一些问题，所以更易于sudo easy_install pip。 （通常，easy_install pip是个不错的主意，它只适用于OS X
10.5-10.8）。另外，10.5-10.8将readline包含在easy_install知道如何使用它，但是pip不是，所以如果你想升级的话，你也需要sudo easy_install readline。

从Ian Bicking自己的[介绍PIP ](https://pip.readthedocs.org/en/1.1/other-tools.html

#pip-compared-to-easy-install)：

pip was originally written to improve on easy_install in the following ways

>

  • All packages are downloaded before installation. Partially-completed
    installation doesnat occur as a result.

  • Care is taken to present useful output on the console.

  • The reasons for actions are kept track of. For instance, if a package is
    being installed, pip keeps track of why that package was required.

  • Error messages should be useful.

  • The code is relatively concise and cohesive, making it easier to use
    programmatically.

  • Packages donat have to be installed as egg archives, they can be
    installed flat (while keeping the egg metadata).

  • Native support for other version control systems (Git, Mercurial and
    Bazaar)

  • Uninstallation of packages.

  • Simple to define fixed sets of requirements and reliably reproduce a set
    of packages.
