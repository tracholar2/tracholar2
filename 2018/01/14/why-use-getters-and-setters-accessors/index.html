为什么使用getter和setter / accessors？ [关闭]

By zhizi

发表于 2018-01-14

文章目录

使用getter和setter的好处是 - 只能获取和设置 - 而不是简单地为这些变量使用公共字段？

如果吸气人员和安装人员做的不仅仅是简单的get / set，我可以很快地把这个问题弄清楚，但是我不是100％清楚的：

public String foo;

比：

更糟糕

private String foo;
public void setFoo(String foo) { this.foo = foo; }
public String getFoo() { return foo; }

而前者需要少量的样板代码。

考虑使用访问器**实际上有很多好的理由_，而不是直接暴露一个类的字段 - 除了封装的参数以及使未来的变化更容易。

以下是我知道的一些原因：

  • 封装与获取或设置属性相关的行为 - 这允许稍后添加更多功能（如验证）。
  • 使用替代表示方式公开属性的内部表示形式。
  • 将您的公共接口从变化中隔离出来 - 允许公共接口在实现发生变化的同时保持不变，而不会影响现有客户。
  • 控制属性的生命周期和内存管理（处置）语义 - 在非托管内存环境（如C ++或Objective-C）中尤为重要。
  • 为属性在运行时更改时提供调试拦截点 - 调试属性更改为特定值的时间和位置可能相当困难，而在某些语言中没有这一点。
  • 改进了与设计为针对属性获取器/设置器运行的库的互操作性 - 让人想起模拟，序列化和WPF。
  • 允许继承者通过重写getter / setter方法来改变属性行为的语义。
  • 允许getter / setter作为lambda表达式而不是值传递。
  • getter和setter可以允许不同的访问级别 - 例如get可能是公开的，但是set可以被保护。

因为从现在起2个星期（几个月，几年），当你意识到你的制定者需要做更多**而不仅仅是设置价值时，你也会意识到这个属性已经被直接用于238其他类： - ）
