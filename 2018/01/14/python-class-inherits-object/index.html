    • 是否有任何理由让类声明继承object？

    我刚刚发现了一些代码，我找不到一个好的理由。

    class MyClass(object):
    # class code follows...

    Is there any reason for a class declaration to inherit from object?

    在Python 3中，除了Python 2和3之间的兼容性外， no 。在Python 2中， 很多原因 。

    Python 2.x story:

    在Python
    2.x中（从2.2开始）有两种类型的类，这取决于是否存在内置类型作为基类：

    ** <预> <代码>＆GT;＆GT;＆GT; class ClassicSpam：＃没有基类 通过 ＆GT;＆GT;＆GT;
    ClassicSpam . bases （）

    style **类：它们有一个内置类型作为基类，它们直接或间接地以object为基类：

    <预> <代码>＆GT;＆GT;＆GT;类NewSpam（object）：＃直接从对象继承 通过 ＆GT;＆GT;＆GT; NewSpam .
    bases     （＆lt; type’object’＆gt ;,） ＆GT;＆GT;＆GT;类IntSpam（int）：＃间接继承对象… 通过
    ＆GT;＆GT;＆GT; IntSpam . bases （＆lt;类型’int’＆gt ;,） ＆GT;＆GT;＆GT; IntSpam .
    bases     [0] . bases＃…因为int从object继承 （＆lt; type’object’＆gt ;,）

    毫无疑问，当你写一个类的时候，你总是希望去学习新的类。这样做的好处很多，列出其中的一些：

    • 支持描述符。具体来说，下面的结构是可能的描述符：

      • classmethod ：一种方法接收该类作为隐式参数而不是实例。
      • staticmethod ：一种方法作为第一个参数没有收到隐式参数self 使用 property 的
      • 属性：创建用于管理属性的获取，设置和删除的功能。
      • __ slots __ ：节省内存消耗的类，也导致更快的属性访问。当然，它会施加限制。 / LI>

    • [ __ new __ 静态方法：让你自定义如何创建新的类实例。

    • 方法解析订单（MRO）：以何种顺序在尝试解析要调用的方法时，将会搜索类的基类。

    • 与MRO相关， 超级调用。另请参阅 super（）认为超级。

    如果你不从object继承，忘记这些。以前的要点以及“新”风格类的其他特性的更详尽的描述可以参见此处。

    新式课程的一个缺点是课程本身对内存的要求更高。除非你创建了许多类对象，否则我怀疑这将是一个问题，这是一个积极的消极的消极影响。

    Python 3.x story:

    在Python 3中，事情被简化了。只有新的类存在（简单地称为类），所以添加object的唯一区别是要求输入8个以上的字符。这样：

    class ClassicSpam:
    pass

    完全相同（除了名称:-)：

    class NewSpam(object):
     pass

    和这个：

    class Spam():
    pass

    在__ bases __中都有object。

    >>> [object in cls.__bases__ for cls in {Spam, NewSpam, ClassicSpam}]
[True, True, True]

    So, what should you I do?

    在Python 2中：** 始终明确地继承object 。获得福利。

    在Python 3中：**继承自object，如果您正在编写试图成为Python不可知的代码，也就是说，它需要在Python
    2和Python否则不要，这真的没有什么区别。

    Python 3.x：
    class MyClass（object）： =新风格的类 class MyClass： =新风格的类（隐式继承对象）

    Python 2.x：
    class MyClass（object）： =新风格的类 class MyClass： = OLD-STYLE CLASS

    Explanation:

    在Python 3.x中定义基类时，您可以从定义中删除对象。但是，这可以打开一个严重的难以追踪问题的大门… …

    Python在Python 2.2中引入了新风格的类，到现在为止，旧风格的类已经很老了。讨论旧式的课程是将[埋在2.x文档中a>，在3.x文档中不存在。

    问题是，<2.x> Python 2.x中旧式类的语法与Python 3.x中新式类的替代语法**相同。 Python
    2.x仍然被广泛使用（例如GAE，Web2Py），任何代码（或者编码器）不知不觉地将2.x风格的类定义带入2.x代码，最终会导致一些严重过期的基础对象。而且因为老式的课程并不是任何人的雷达，他们可能不知道是什么打击了他们。

    所以，只要把它拼出来，节省一些2.x开发人员的眼泪。
