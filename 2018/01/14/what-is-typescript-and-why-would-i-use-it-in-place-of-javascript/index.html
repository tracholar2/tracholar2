    TypeScript是JavaScript的超集，主要提供可选的静态类型，类和接口。其中一个最大的好处就是可以让IDE提供一个更加丰富的环境，以便在输入代码的同时发现常见的错误

    对于一个大型的JavaScript项目，采用TypeScript可能会产生更强大的软件，同时仍然可以在普通的JavaScript应用程序运行的地方部署。

    它是开放源代码的，但是如果您使用受支持的IDE，您只能在键入时获得智能的Intellisense。最初，这只是微软的Visual Studio（也从[Miguel de Icaza < /

    Are there other technologies like it?

    有 CoffeeScript，但这确实起到了不同的作用。恕我直言，CoffeeScript为人类提供了可读性，但是TypeScript还通过其可选的静态类型为

    Example

    举个例子，下面是一些TypeScript（你可以在 TypeScript Playground上玩这个） / p>

    class Greeter {
    greeting: string;
    constructor (message: string) {
        this.greeting = message;
    }
    greet() {
        return "Hello, " + this.greeting;
    }
}

    这是它会产生的JavaScript

    var Greeter = (function () {
    function Greeter(message) {
        this.greeting = message;
    }
    Greeter.prototype.greet = function () {
        return "Hello, " + this.greeting;
    };
    return Greeter;
})();

    注意TypeScript定义了成员变量和类方法参数的类型。这在转换为JavaScript时被删除，但被IDE和编译器用来发现错误，比如将数字类型传递给构造函数。

    它还能够推断未明确声明的类型，例如，它将确定greet（）方法返回一个字符串。

    Want to know more?

    我最初写这个答案时，Typescript仍然热门新闻。查看这个问题的Lodewijk的答案以获得更多的当前细节。

    虽然被接受的答案是好的，但是我觉得在这一点上确实没有TypeScript正义。现在不是早期了。现在TypeScript正在用TypeScript编写几个流行的框架，现在发现更多的采用。现在你应该选择TypeScript而不是JavaScript。

    Relation to JavaScript

    JavaScript通过ECMAScript标准进行了标准化。并非所有正在使用的浏览器都支持新的ECMAScript标准的所有功能（请参阅此表）。
    TypeScript支持新的ECMAScript标准，并将它们编译为您选择的（较旧的）ECMAScript目标（当前目标是3,5和6 [a.k.a.
    2015]）。这意味着您可以使用ES2015及更高版本的功能，比如今天的模块，lambda函数，类，扩展运算符，解构。它还增加了当然的类型支持，这不是任何ECMAScript标准的一部分，并且可能永远不会由于解释性而不是JavaScript的编译本质。
    TypeScript的类型系统相对丰富，包括：接口，枚举，混合类型，泛型，联合和交集类型，访问修饰符等等。
    TypeScript的官方网站概述了这些功能。

    Relation to other JavaScript targeting languages

    与其他编译成JavaScript的语言相比，TypeScript有一个独特的哲学。 JavaScript代码是有效的TypeScript代码;
    TypeScript是JavaScript的超集。您几乎可以将您的.js文件重命名为.ts文件，并开始使用TypeScript。
    TypeScript文件被编译为可读的JavaScript，因此可以回迁，理解已编译的TypeScript并不难。通过这种方式，TypeScript建立在JavaScript的成功之上，同时改进了其弱点。

    一方面，您有将来的验证工具，采用现代的ECMAScript标准，并将其编译成较旧的JavaScript版本，Babel是最流行的版本。另一方面，您的语言可能完全不同于JavaScript，比如Coffeescript，Clojure，Dart，Elm，Haxe，ScalaJs以及更多的整个主机（详见list）。这些语言虽然可能比JavaScript的未来可能更好，但更有可能没有找到足够的采用来保证它们的未来。对于这些语言中的某些语言，您可能会遇到更多麻烦，尽管您会发现这些语言通常会更热情。使用JavaScript进行互操作也可能会涉及更多，因为它们远离JavaScript的实际内容。

    TypeScript坐在这两个极端之间，从而平衡风险。
    TypeScript不是任何标准的风险选择。如果您熟悉JavaScript，只需要花费很少的精力，因为它不是一种完全不同的语言，具有出色的JavaScript互操作性支持，并且近来已经被大量采用。

    Optionally static typing and type inference

    JavaScript是动态输入的。这意味着JavaScript不知道什么类型的变量，直到它实际上在运行时实例化。这也意味着它可能为时已晚。
    TypeScript为JavaScript添加了类型支持。错误地假设某些变量是某种类型的错误，如果你正确地玩牌，可以彻底根除;你输入你的代码有多严格，或者你输入你的代码完全取决于你。

    TypeScript通过类型推断的使用使打字变得简单一些，而不那么明显。例如：TypeScript中的var x =“hello”与var x：string =“hello”相同。这种类型只是从它的使用推断。即使你没有明确地键入类型，他们仍然在那里，以避免做一些事情，否则会导致运行时错误。

    TypeScript默认是可选的。例如函数divideByTwo（x）{return x / 2}是TypeScript中的一个有效函数，可以用
    任何类型的参数调用，即使调用字符串显然会导致一个 运行时_
    错误。就像你习惯在JavaScript中一样。这是有效的，因为当没有明确指定类型并且无法推断类型时（比如在divideByTwo示例中），TypeScript将隐式地指定类型any。这意味着divideByTwo函数的类型签名自动成为function divideByTwo（x：any）：any。有一个编译器标志来禁止这种行为：- noImplicitAny。启用此标志可以提高您的安全性，但也意味着您需要进行更多的输入。

    类型有一个与他们相关的成本。首先有一条学习曲线，其次，当然，使用适当的严格打字也需要花费更多的时间来设置代码库。根据我的经验，这些代价是完全值得的任何严重的代码库，你与他人分享。
    Github编程语言和代码质量的大规模研究表明，静态类型的语言通常比动态类型更不容易出现缺陷，在同样的方面，强类型优于弱类型。“_

    Enhanced IDE support

    使用TypeScript的开发经验是JavaScript的一大改进。
    IDE通过TypeScript编译器实时通知其丰富的类型信息。这提供了几个主要的优点。例如，使用TypeScript，您可以安全地重构整个代码库中的重命名。通过代码完成，您可以获得图书馆可能提供的任何函数的内联帮助。没有更多需要记住他们或在网上查阅他们。编译错误直接在IDE中用红色波浪线报告，而您正忙于编码。总之，与使用JavaScript相比，这样做可以显着提高工作效率。人们可以花更多的时间进行编码，调试时间更少。

    有很多IDE对TypeScript有很好的支持，比如Visual Studio＆amp; VS代码，Atom，Sublime和IntelliJ /
    WebStorm。

    Strict null checks

    形式的运行时错误不能读取undefined
    的属性'x'undefined不是函数通常是由JavaScript代码中的错误引起的。由于不能使用TypeScript编译器不知道的变量（任何类型变量）的属性），TypeScript已经可以减少发生这类错误的可能性。尽管错误地使用设置为undefined的变量仍然是可能的。但是，使用TypeScript的2.0版本，您可以通过使用不可空的类型来消除这些类型的错误。这工作如下：

    如果启用严格的空检查（- strictNullChecks编译器标志），TypeScript编译器将不允许将undefined分配给变量，除非您显式声明它为可空类型。例如，let x：number = undefined会导致编译错误。这符合类型理论，因为undefined不是一个数字。可以将x定义为number和undefined的和类型来解决此问题：let x：number | undefined = undefined。简短的符号是let x：number？ =未定义。

    一旦一个类型被认为是可空的，意味着它是一个类型，也可以是null或undefined的值，TypeScript编译器可以通过控制基于流的类型分析是否您的代码可以安全地使用变量或不。换句话说，当通过例如if语句检查一个变量是undefined时，TypeScript编译器会推断出你的代码控制流的该分支中的类型不再是可以为空的因此可以安全地使用。这是一个简单的例子：

    let x : number?;
if (x !== undefined) x += 1; // this line will compile, because x is checked.
x += 1; // this line will fail compilation, because x might be undefined.

    Compilation

    要使用TypeScript，你需要一个编译过程来编译成JavaScript代码。构建过程通常只需要几秒钟，具体取决于项目的大小。
    TypeScript编译器支持增量编译（- watch编译器标记），以便所有后续更改可以更快地编译。

    TypeScript编译器可以在生成的.js文件中内联源地图信息或创建单独的.map文件。源代码映射信息可以被像Chrome
    DevTools和其他IDE这样的调试工具用来将JavaScript中的行与在TypeScript中生成的代码联系起来。这使您可以在运行时直接在TypeScript代码上设置断点和检查变量。源地图信息工作得相当不错，大约在TypeScript之前，但是调试TypeScript通常不如直接使用JavaScript那么好。以this关键字为例。由于自ES2015以来围绕this关键字的this关键字发生了改变，这个实际上可能在运行时存在为一个名为_this的变量（参见这个答案）。在调试过程中，这可能会让你感到困惑，但是如果你知道这个问题或者检查JavaScript代码，通常不会有问题。应该指出的是，巴贝尔遭受同样的问题。

    还有一些TypeScript编译器可以做的其他技巧，比如根据装饰器生成拦截代码，生成模块加载代码为不同的模块系统和解析JSX。但是，除了Typescript编译器之外，您可能还需要构建工具。例如，如果你想压缩你的代码，你将不得不在你的构建过程中添加其他工具来完成。

    有 Webpack 的TypeScript编译插件， Gulp， Grunt 和几乎所有其他的JavaScript构建工具。
    TypeScript文档中有关于与构建工具集成的章节涵盖所有。 linter也可以用来进行更多的编译时间检查。还有大量的种子项目，可以让你开始使用TypeScript和Angular
    2，React，Ember，SystemJs，WebPack，Gulp等其他一些技术。

    JavaScript interoperability

    由于TypeScript与JavaScript密切相关，因此具有良好的互操作性，但是在TypeScript中使用JavaScript库需要额外的工作。需要使用TypeScript定义，以便TypeScript编译器能够理解像_.groupBy或angular.copy或$。fadeOut实际上不是非法的语句。这些函数的定义放置在.d.ts文件中。

    定义最简单的形式是允许以任何方式使用标识符。例如，当使用 Lodash 时，单行定义文件declare var _：any将允许你可以在_上调用你想要的任何函数，当然你也可以犯错：_。foobar（）是合法的TypeScript调用，在运行时当然是非法的。如果你想要正确的类型支持和代码完成你的定义文件需要更确切（见lodash定义为例）。

    由TypeScript编译器自动理解Npm模块与其自己的类型定义一起预先打包（请参阅文档）。对于几乎所有其他半流行的JavaScript库，它都没有包含自己的定义（目前大多数项目），那里的某个人已经创建了.d.ts文件。可以使用名为类型的命令行工具下载和管理这些文件。类型的大部分类型定义都来自一个叫做DefinitelyTyped的Github仓库，但是也可以通过其他来源如Github，Npm，Bower等。安装类型之后，您可以在项目目录的命令行上运行typings install lodash，您就可以开始使用Lodash了。现在，当您在编辑器中键入_。时，您的IDE会准确告诉您哪些Lodash函数可用。

    有一个警告：TypeScript定义必须与您在运行时使用的库的版本相匹配。如果他们不这样做，TypeScript可能会禁止您调用函数或取消引用存在的变量，或者允许您调用函数或取消引用不存在的变量。类型使用注册表能够提供不同版本的定义。这是一个相当新的发展，说实话，还没有很多不同版本的库可用，所以通常情况下，最好使用库的最新版本，因为定义往往是最新的。
    / p>
    老实说，这有点麻烦，这可能是你不选择TypeScript的原因之一，而是像Babel这样的东西，根本不需要得到类型定义。另一方面，如果你知道你在做什么，你可以轻松地克服任何类型的错误或缺少定义文件的问题。

    Converting from JavaScript to TypeScript

    任何.js文件都可以重命名为.ts，并通过TypeScript编译器在语法上得到与输出相同的JavaScript代码（如果它在语法上是正确的首先）。即使TypeScript编译器遇到编译错误，它仍然会产生一个.js文件。它甚至可以接收.js文件作为- allowJs标志的输入。这使您可以马上开始使用TypeScript。不幸的是编译错误很可能在一开始就发生。其中一个确实需要记住，这些并不像你可能习惯的其他编译器那样显示停止错误。

    在将JavaScript项目转换为TypeScript项目时，开始的编译错误是TypeScript本质所无法避免的。
    TypeScript检查所有的_代码的有效性，因此它需要知道所有使用的函数和变量。因此类型定义需要适用于所有这些类型，否则编译错误必然会发生。正如在上面的章节中提到的，几乎任何JavaScript框架都有.d.ts文件，可以通过几个类型install命令轻松获取。但是，可能是因为您使用了一些没有TypeScript定义可用的模糊库，或者您已经填充了一些JavaScript基元。在这种情况下，您必须为这些位提供类型定义，以免编译错误消失。只需创建一个.d.ts文件，并将其包含在tsconfig.json的files数组中，以便TypeScript编译器始终考虑它。在其中声明TypeScript不知道的那些位是any类型。一旦你消除了所有的错误，你可以根据你的需要，逐渐将这些部分输入到这些部分。

    一些工作（重新）配置您的生成管道也将需要将TypeScript到生成管道。正如在编译章节中提到的那样，我们鼓励大家寻找使用你想要工作的工具组合的种子项目。

    最大的障碍是学习曲线。我鼓励你先玩一个小项目。看看它是如何工作的，如何构建，使用哪些文件，如何配置，如何在IDE中运行，如何构建，使用哪些工具等。将大型JavaScript代码库转换为TypeScript是非常可行的知道你在做什么，但是如果你不这样做，可能会感到沮丧。
