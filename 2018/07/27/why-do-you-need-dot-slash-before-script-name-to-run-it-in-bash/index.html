为什么你需要 /（点斜线）之前的脚本名称运行在bash？

By zhizi

Published 2018-07-27

Contents

在bash中运行脚本时，我必须在开头写./：

$ ./manage.py syncdb

如果我不这样做，我会收到错误消息：

$ manage.py syncdb
-bash: manage.py: command not found

这是什么原因？我以为。是当前文件夹的别名，因此这两个调用应该是等效的。

我也不明白为什么我在运行应用程序时不需要./，比如：

user:/home/user$ cd /usr/bin
user:/usr/bin$ git

（运行时没有./）

因为在Unix上，通常当前目录不在$ PATH中。

当你输入一个命令时，shell会按照PATH变量的指定查找一个目录列表。当前目录不在该列表中。

没有当前目录的理由是安全。

假设你是root用户，进入另一个用户的目录，输入sl而不是ls。如果当前目录在PATH中，那么shell将尝试执行该目录下的sl程序（因为没有其他的sl程序） 。这个sl程序可能是恶意的。

它适用于./，因为POSIX指定包含/的命令名称将被直接用作文件名，从而禁止在$ PATH中进行搜索。您可以使用完整路径来获得完全相同的效果，但是./更短且更容易编写。

**修改

sl部分仅仅是一个例子。 PATH中的目录是按顺序搜索的，当匹配时执行程序。因此，根据PATH的外观，输入一个正常的命令可能会或可能不足以在当前目录中运行程序。

当bash解释命令行时，它会在环境变量$ PATH中描述的位置查找命令。要看到它，请键入：

echo $PATH

你会有一些冒号分隔的冒号。您将会看到当前路径。通常不在$ PATH中。所以Bash找不到你的命令，如果它在当前目录下。你可以改变它：

PATH=$PATH:.

这行添加了$ PATH中的当前目录，所以你可以这样做：

manage.py syncdb

不推荐使用 ** ，因为它存在安全问题，另外您可能会有奇怪的行为，因为。会根据您所在的目录而变化：）

避免：

PATH=.:$PATH

正如你可以“掩盖”一些标准命令并打开安全漏洞的大门:)
