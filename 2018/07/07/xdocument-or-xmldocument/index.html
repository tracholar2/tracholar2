XDocument或XmlDocument

By zhizi

Published 2018-07-07

Contents

我现在正在学习 XmlDocument 但我刚刚遇到 XDocument，当我尝试搜索他们的差异或好处，我找不到有用的东西，请告诉我为什么你会使用一个在另一个吗？

如果您使用的是.NET 3.0或更低版本，则可以使用XmlDocument（也就是经典的DOM
API）。同样，你会发现还有其他一些API可以期待这一点。

然而，如果你有选择的话，我会建议使用XDocument，也就是LINQ to XML。创建文档和处理它们非常简单。例如，这是区别：

XmlDocument doc = new XmlDocument();
XmlElement root = doc.CreateElement("root");
root.SetAttribute("name", "value");
XmlElement child = doc.CreateElement("child");
child.InnerText = "text node";
root.AppendChild(child);
doc.AppendChild(root);

和

XDocument doc = new XDocument(
    new XElement("root",
                 new XAttribute("name", "value"),
                 new XElement("child", "text node")));

命名空间很容易在LINQ to XML中使用，与我见过的任何其他XML API不同：

XNamespace ns = "http://somewhere.com";
XElement element = new XElement(ns + "elementName");
// etc

LINQ to XML对于LINQ也起到了很好的作用 - 它的构建模型允许你很容易地构建带有子元素序列的元素：

// Customers is a List<Customer>
XElement customersElement = new XElement("customers",
    customers.Select(c => new XElement("customer",
        new XAttribute("name", c.Name),
        new XAttribute("lastSeen", c.LastOrder)
        new XElement("address",
            new XAttribute("town", c.Town),
            new XAttribute("firstline", c.Address1),
            // etc
    ));

它更具说明性，符合一般的LINQ风格。

正如Brannon所说，这些是内存中的API，而不是流式API（尽管XStreamingElement支持延迟输出）。 XmlReader和XmlWriter是在.NET中流式传输XML的常用方式，但是您可以在一定程度上混合使用所有的API。例如，可以通过在元素的开始处定位XmlReader，从中读取XElement并处理它，然后移动一个大文档，但使用LINQ to
XML到下一个元素等。有关于这种技术的各种博客文章，这是我通过快速搜索找到的一个。

我感到惊讶迄今为止没有任何答案提到 XmlDocument不提供行信息 ，而 **XDocument / strong>（通过<代码> IXmlLineInfo ` 接口）。

在某些情况下，这可能是一个重要的功能（例如，如果您想要在XML中报告错误，或者跟踪通常定义的元素的位置），并且在开始使用之前，您最好意识到这一点XmlDocument，以后发现你必须改变它。
