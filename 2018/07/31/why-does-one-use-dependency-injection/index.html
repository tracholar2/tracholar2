为什么要使用依赖注入？ [关闭]

By zhizi

发表于 2018-07-31

文章目录

我正在尝试了解依赖项注入（DI），而我又一次失败了。这似乎很愚蠢。我的代码从来不是一团糟。我很少编写虚拟函数和接口（虽然我只做了一次蓝月亮），所有的配置都被神奇地用json.net（有时使用XML序列化程序）序列化成一个类。

我不太明白它解决了什么问题。它看起来像一个方式来说：“嗨，当你遇到这个函数，返回一个这种类型的对象，并使用这些参数/数据。”
但是…为什么我会用这个？注意我从来也不需要使用object，但是我明白这是什么意思。

在使用DI的网站或桌面应用程序中，有哪些实际情况？为了让人们可以在游戏中使用接口/虚函数，我可以很容易地想出一些例子，但是在非游戏代码中使用非常罕见（难以记忆单个实例）是非常罕见的。
p> 首先，我想解释一个假设，我为这个答案。这并非总是如此，而是经常发生：

Interfaces are adjectives; classes are nouns.

（实际上，也有名词的界面，但我想在这里概括一下。）

所以，例如接口可能是诸如IDisposable，IEnumerable或IPrintable之类的东西。一个类是一个或多个这些接口的实际实现：List或者Map都可以是IEnumerable的实现。

为了说明问题：通常你们的课程是相互依赖的。例如。你可以有一个访问你的数据库的Database类（hah，surprise！;-)），但是你也希望这个类来做访问数据库的日志记录。假设您有另一个类Logger，那么Database对Logger有依赖关系。

到目前为止，这么好。

您可以使用以下行在您的Database类中对此依赖关系进行建模：

var logger = new Logger();

一切都很好。直到你意识到你需要一堆记录器的时候，这是没问题的：有时你想登录到控制台，有时到文件系统，有时使用TCP / IP和远程日志记录服务器，等等。

当然，你不是要改变你所有的代码（同时你有它的代码）并且替换所有的行

var logger = new Logger();

由：

var logger = new TcpLogger();

首先，这并不好玩。其次，这是容易出错的。第三，对于训练有素的猴子来说，这是愚蠢的，重复的工作。那么你做什么？

显然，引入一个由所有各种记录器实现的接口ICanLog（或类似的）是个不错的主意。所以你的代码中的第1步是你：

ICanLog logger = new Logger();

现在类型推断不再改变类型，你总是有一个单一的界面发展反对。下一步是你不想一次又一次地使用new Logger（）。所以你把可靠性创建到一个单一的中央工厂类，你得到的代码如：

ICanLog logger = LoggerFactory.Create();

工厂自己决定创建什么样的记录器。你的代码不再关心，如果你想改变使用的记录器的类型，你可以改一次 ： ：在工厂里面。

现在，当然，你可以概括这个工厂，并使其适用于任何类型：

ICanLog logger = TypeFactory.Create<ICanLog>();

在某个地方，这个TypeFactory需要配置数据，当请求一个特定的接口类型时，实际的类将被实例化，所以你需要一个映射。当然，你可以在代码中做这个映射，但是类型改变意味着重新编译。但是你也可以把这个映射放在一个XML文件中，例如..这使得你可以改变实际使用的类，即使在编译时（！），这意味着动态的，而不需要重新编译！

给你一个有用的例子：想想一个不能正常记录的软件，但是当你的客户打电话求助，因为他有问题时，你发给他的是一个更新的XML配置文件，现在他启用了日志记录，并且您的支持可以使用日志文件来帮助您的客户。

现在，当您替换一些名称时，您最终会实现一个 Service Locator ，它是 控制反转 的两种模式之一。
（因为你不能控制谁决定实例化哪个类）。

总而言之，这减少了代码中的依赖关系，但现在所有的代码都依赖于中央的单一服务定位器。

_现在是这一行的下一个步骤：只要摆脱这个单一的依赖到服务定位器：而不是各种类的要求服务定位器的具体接口的实现，你 - 再一次 - 重新控制谁实例化什么。

使用依赖注入，你的Database类现在有一个构造函数，它需要一个ICanLog类型的参数：

public Database(ICanLog logger) { ... }

现在你的数据库总是有一个记录器来使用，但它不知道这个记录器来自哪里。

这就是DI框架的作用：你再次配置你的映射，然后让你的DI框架为你实例化你的应用程序。由于Application类需要一个ICanPersistData实现，所以注入Database的一个实例，但是为此它必须首先创建一个实例为ICanLog配置的记录器。等等…

所以，简而言之，简单来说：依赖注入是如何去除代码中依赖关系的两种方法之一。这对于编译后的配置更改非常有用，对于单元测试来说是一件好事（因为它使注入存根和/或模拟变得非常简单）。

在实践中，没有服务定位器是不能做的事情（例如，如果事先不知道你需要多少个实例需要一个特定的接口：一个DI框架每个参数总是只注入一个实例，但是当然，你可以在一个循环中调用一个服务定位器），因此大多数情况下每个DI框架都提供一个服务定位器。

但基本上就是这样。

希望有帮助。

PS：我在这里描述的是一种叫做 构造函数注入 的技术，还有 属性注入
，其中不是构造函数参数，而是用于定义和解析相关性的属性。将属性注入视为可选的依赖项，将构造函数注入视为必需的依赖项。但是这个问题的讨论超出了这个问题的范围。

我认为很多时候人们对于依赖注入和依赖注入框架之间的区别感到困惑，因为它是一个容器_经常被称为）。

依赖注入是一个非常简单的概念。而不是这个代码：

public class A {
  private B b;

  public A() {
    this.b = new B(); // A *depends on* B
  }

  public void DoSomeStuff() {
    // Do something with B here
  }
}

public static void Main(string[] args) {
  A a = new A();
  a.DoSomeStuff();
}

你写这样的代码：

public class A {
  private B b;

  public A(B b) { // A now takes its dependencies as arguments
    this.b = b; // look ma, no "new"!
  }

  public void DoSomeStuff() {
    // Do something with B here
  }
}

public static void Main(string[] args) {
  B b = new B(); // B is constructed here instead
  A a = new A(b);
  a.DoSomeStuff();
}

就是这样。**真的。这给了你很多好处。两个重要的是能够从一个中心位置（Main（）函数）控制功能，而不是将其分散到整个程序中，并且更容易地单独测试每个类（因为可以将模拟对象或其他伪造的对象传递给它的构造函数，而不是真正的值）。

当然，缺点是你现在有一个超级函数可以知道程序使用的所有类。这是DI框架可以帮助的。但是如果你不明白为什么这个方法是有价值的，我建议先从手动依赖注入入手，这样你就能更好地理解那里的各种框架能为你做些什么。
