在C＃中使用var关键字

By zhizi

发表于 2018-07-19

文章目录

在与同事讨论了在C＃3中使用“var”关键字之后，我想知道人们对var类型推断的适当用法有何看法？

例如，我比较懒惰地在可疑的情况下使用var，例如： -

foreach(var item in someList) { // ... } // Type of 'item' not clear.
var something = someObject.SomeProperty; // Type of 'something' not clear.
var something = someMethod(); // Type of 'something' not clear.

var更合理的用法如下：

var l = new List<string>(); // Obvious what l will be.
var s = new SomeClass(); // Obvious what s will be.

有趣的是，LINQ似乎有点灰色地带，例如： -

var results = from r in dataContext.SomeTable
              select r; // Not *entirely clear* what results will be here.

很明显，结果将会是一个实现了IEnumerable的类型，但是它和声明一个新对象的var不一样。

LINQ到对象更糟，例如： -

var results = from item in someList
              where item != 3
              select item;

这并不比等同的foreach（在someList中的var项）// // …}等同。

这里有一个关于类型安全性的真正问题 - 例如，如果我们将查询的结果放入一个接受IEnumerable＆lt;
int＆gt;的重载方法中，和IEnumerable＆lt; double＆gt;调用者可能会无意中传入错误的类型。

var ` does
保持强类型，但问题是这样做的危险在于定义时不能立即显示类型，当重载意味着编译器错误当您无意中将错误的类型传递给某个方法时，可能不会被发布。

我仍然认为在某些情况下，var可以使代码更具可读性。如果我有一个带有订单属性的客户类，并且我想把它分配给一个变量，我会这样做：

var orders = cust.Orders;

我不在乎Customer.Orders是否为IEnumerable＆lt; Order＆gt;，ObservableCollection＆lt; Order＆gt;或BindingList＆lt; Order＆gt;我想要的是保留在内存中的列表来迭代它或得到它的计数或稍后的东西。

将上述声明与

对比

ObservableCollection<Order> orders = cust.Orders;

对我来说，类型名称只是噪音。如果我回去决定改变Customer.Orders的类型沿轨道（从ObservableCollection＆lt; Order＆gt; to IList＆lt; Order＆gt;），那么我需要也改变这个声明 - 如果我在第一个地方使用了var，我就不必做了。

我广泛使用var。有人批评说，这会降低代码的可读性，但没有理由支持这种说法。

不得不承认，这可能意味着我们不清楚我们正在处理的是什么类型。所以呢？这实际上是一个分离设计的重点。在处理接口的时候，你强调 not 对变量的类型感兴趣。
var更进一步说明了这一点，但是我认为，从可读性的角度来看，参数仍然是一样的：程序员实际上不应该对变量的类型感兴趣，变量 不是
。这就是为什么微软也会调用类型推断“打字输入”的原因 那么，当我使用var声明变量时，变量会做什么？简单来说，IntelliSense告诉我它确实做了什么。任何关于C＃忽略IDE的推理都不符合现实。实际上，每个C＃代码都在支持IntelliSense的IDE中进行编程。

如果我正在使用一个var声明的变量，并且弄糊涂了那个变量，那么我的代码就有一些根本性的错误。 var不是原因，只会使症状可见。不要责怪信使。

现在，C＃团队发布了一个编码指南，指出var仅用于捕获创建匿名类型的LINQ语句的结果（因为这里，我们没有真正的替代var）。那么，拧紧。只要C＃团队没有给我这个指导方针一个合理的论点，我会忽略它，因为在我的专业和个人的意见，这是纯粹的胡扯。
（对不起，我没有链接到有关指南。）

实际上，有一些（表面上）很好的解释为什么你不应该使用var，但是我仍然认为它们在很大程度上是错误的。以“可检索性”为例：作者声称var使得很难搜索使用MyType的地方。对。所以做接口。其实，为什么我想知道这个班在哪里？我可能更感兴趣的是它在哪里实例化，这仍然是可搜索的，因为它的构造函数必须被调用（即使这是间接完成的，类型名称必须在某处提及）。
