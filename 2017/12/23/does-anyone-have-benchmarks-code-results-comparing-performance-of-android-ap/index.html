    • 我遇到了Xamarin声称他们在Android和他们的C＃编译的应用程序上的Mono实现比Java代码更快。有没有人在不同的Android平台上对非常相似的Java和C＃代码进行实际的基准测试来验证这些声明，可以发布代码和结果？

    Added June 18, 2013

    既然没有答案，也找不到别人所做的基准，就决定做我自己的测试。不幸的是，我的问题仍然是“锁定”，所以我不能发表这个答案，只能编辑问题。请投票重新提出这个问题。对于C＃，我使用Xamarin.Android版本。
    4.7.09001（测试版）源代码，用于测试的所有数据以及已编译的APK软件包位于GitHub上：

    Java：https：//github.com/gregko/TtsSetup_Java

    C＃：https：//github.com/gregko/TtsSetup_C_sharp

    如果有人想在其他设备或模拟器上重复我的测试，我也有兴趣了解结果。

    Results from my testing

    我将我的句子提取器类移植到C＃（从我的@语音朗读器应用程序），并对英语，俄语，法语，波兰语和捷克语的10个html文件运行一些测试。每个运行在所有10个文件上执行5次，下面列出3个不同设备和一个模拟器的总时间。我只测试了“Release”版本，没有启用调试。

    HTC Nexus One Android 2.3.7 (API 10) - CyanogenMod ROM

    Java：总计时间（5次）：12361毫秒，文件读取总数：13304毫秒

    C＃：总计时间（5次）：17504毫秒，文件读取总计：17956毫秒

    Samsung Galaxy S2 SGH-I777 (Android 4.0.4, API 15) - CyanogenMod ROM

    Java：总时间（5次）：8947毫秒，文件读取总数：9186毫秒

    C＃：总时间（5次）：9884毫秒，文件读取总数：10247毫秒

    Samsung GT-N7100 (Android 4.1.1 JellyBean, API 16) - Samsung ROM

    Java：总时间（5次）：9742毫秒，文件读取总数：10111毫秒

    C＃：总计时间（5次）：10459毫秒，文件读取总数：10696毫秒

    Emulator - Intel (Android 4.2, API 17)

    Java：总计时间（5次）：2699毫秒，文件读数总计：3127毫秒

    C＃：总计时间（5次）：2049毫秒，文件读取总计：2182毫秒

    Emulator - Intel (Android 2.3.7, API 10)

    Java：总计时间（5次）：2992ms，文件读取总数：3591ms

    C＃：总时间（5次运行）：2049毫秒，文件读数总计：2257毫秒

    Emulator - Arm (Android 4.0.4, API 15)

    Java：总计时间（5次）：41751毫秒，文件读取总数：43866毫秒

    C＃：累计总时间（5次）：44136毫秒，文件读数总计：45109毫秒

    Brief discussion

    我的测试代码主要包含文本解析，替换和正则表达式搜索，也许对于其他代码（例如更多数字操作），结果将会不同。在所有使用ARM处理器的设备上，Java比Xamarin
    C＃代码表现得更好。最大的区别是在Android 2.3下，C＃代码运行在约。 Java速度的70％。

    在英特尔仿真器上（使用英特尔HAX技术，仿真器以快速虚拟模式运行），Xamarin C＃代码运行我的示例代码比Java快得多 -
    约快1.35倍。也许单声道虚拟机代码和库在英特尔上比在ARM上更好地优化？

    Edit July 8, 2013

    我刚刚安装了运行在Oracle VirtualBox中的Genymotion
    Android模拟器，并且再次使用本机Intel处理器，而不是仿真ARM处理器。与英特尔HAX仿真器一样，C＃在这里运行得更快。这是我的结果：

    Genymotion emulator - Intel (Android 4.1.1, API 16)

    Java: Grand total time (5 runs): 2069 ms, with file reading total: 2248 ms

    >

    C#: Grand total time (5 runs): 1543 ms, with file reading total: 1642 ms

    然后我注意到有一个更新Xamarin.Android测试版，版本4.7.11，发行说明提及Mono运行时的一些变化。决定快速测试一些ARM设备，并大吃一惊

    • C＃编号改进：

    BN Nook XD+, ARM (Android 4.0)

    Java: Grand total time (5 runs): 8103 ms, with file reading total: 8569 ms

    >

    C#: Grand total time (5 runs): 7951 ms, with file reading total: 8161 ms

    哇！ C＃现在比Java更好吗？决定在我的Galaxy Note 2上重复测试：

    Samsung Galaxy Note 2 - ARM (Android 4.1.1)

    Java: Grand total time (5 runs): 9675 ms, with file reading total: 10028 ms

    >

    C#: Grand total time (5 runs): 9911 ms, with file reading total: 10104 ms

    这里C＃似乎只是稍微慢一些，但这些数字给了我一个暂停：为什么时间比Nook HD +更长，尽管Note
    2有一个更快的处理器？答案是：省电模式。在角落，它被禁用，在注2 - 启用。决定在禁用省电模式的情况下进行测试（启用时也会限制处理器速度）：

    Samsung Galaxy Note 2 - ARM (Android 4.1.1), power saving disabled

    Java: Grand total time (5 runs): 7153 ms, with file reading total: 7459 ms

    >

    C#: Grand total time (5 runs): 6906 ms, with file reading total: 7070 ms

    现在，令人惊讶的是，C＃在ARM处理器上也比Java稍快。大改进！

    Edit July 12, 2013

    我们都知道，没有什么比原生代码更快的速度，而且我对Java或C＃中的语句分割器的性能还不满意，特别是我需要改进它（并且因此使其更慢）。决定用C
    ++重新编写它。这是一个小的（比以前的测试文件更小的一组文件，因为其他原因）我的Galaxy Note 2的本机和Java的速度比较，禁用了省电模式：

    Java的： 总时间（5次）：3292毫秒，文件读数总计：3454毫秒

    原生拇指： 总时间（5次）：537毫秒，文件读数总计：657毫秒

    原住民手臂： 总时间（5次）：458毫秒，文件读数总计：587毫秒

    看起来像我的特定测试，本机代码比Java快6到7倍。警告：不能在Android上使用std ::
    regex类，所以不得不编写我自己的搜索段落分隔符或html标签的特殊例程。我使用正则表达式在PC上对同一代码进行的初始测试比Java快4到5倍。

    唷！用char 或wchar 指针再次唤醒原始内存，我立刻感觉到了20岁！ :)

    Edit July 15, 2013

    （请参阅下文，编辑2013年7月30日，与Dot42更好的结果）

    有一些困难，我试图把我的C＃测试移植到Dot42（版本1.0.1.71测试版），Android的另一个C＃平台。初步结果显示，在英特尔Android仿真器上，Dot42代码比Xamarin
    C＃（v.4.7.11）慢3倍（3倍）。一个问题是，Dot42中的System.Text.RegularExpressions类没有在Xamarin测试中使用的Split（）函数，所以我使用了Java.Util.Regex类，而使用了Java.Util.Regex.Pattern.Split
    ），所以在代码中的这个特定的地方有这个小的区别。应该不是一个大问题。
    Dot42编译为Dalvik（DEX）代码，所以它本身就和Android上的Java配合，不需要像Xamarin那样从C＃到Java的昂贵的interop。

    仅仅为了比较，我也在ARM设备上运行测试 - 这里Dot42代码比Xamarin C＃只“慢”2倍。这是我的结果：

    HTC Nexus One Android 2.3.7 (ARM)

    Java: Grand total time (5 runs): 12187 ms, with file reading total: 13200 ms

    >

    Xamarin C#: Grand total time (5 runs): 13935 ms, with file reading total:
    14465 ms

    >

    Dot42 C#: Grand total time (5 runs): 26000 ms, with file reading total:
    27168 ms

    Samsung Galaxy Note 2, Android 4.1.1 (ARM)

    Java: Grand total time (5 runs): 6895 ms, with file reading total: 7275 ms

    >

    Xamarin C#: Grand total time (5 runs): 6466 ms, with file reading total:
    6720 ms

    >

    Dot42 C#: Grand total time (5 runs): 11185 ms, with file reading total:
    11843 ms

    Intel emulator, Android 4.2 (x86)

    Java: Grand total time (5 runs): 2389 ms, with file reading total: 2770 ms

    >

    Xamarin C#: Grand total time (5 runs): 1748 ms, with file reading total:
    1933 ms

    >

    Dot42 C#: Grand total time (5 runs): 5150 ms, with file reading total: 5459
    ms

    对我来说，有趣的是在新的ARM设备上Xamarin C＃比Java稍微快一些，而旧版的Nexus
    One稍慢一些。如果有人想要运行这些测试，请让我知道，我会更新GitHub上的来源。使用英特尔处理器的真实Android设备的结果将是特别有趣的。

    Update 7/26/2013

    只是一个快速更新，由基准应用程序与最新的Xamarin.Android 4.8，以及今天发布的dot42 1.0.1.72更新重新编译 -
    没有从以前的报告结果的重大变化。

    Update 7/30/2013 - better results for dot42

    用我的Java代码的Robert（从dot42制造商）端口重新测试Dot42到C＃。在我最初为Xamarin完成的C＃端口中，我用一些本地Java类（如ListArray）和C＃原生的List类替换了它们。Robert没有我的Dot42源代码，所以他再次从Java移植它，并使用原始Java类这些地方，有利于Dot42，我猜是因为它运行在Dalvik虚拟机，像Java，而不是单声道，如Xamarin。现在Dot42的结果好多了。以下是我测试的日志：
