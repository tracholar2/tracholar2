如何在Unix控制台或Mac终端上运行shell脚本？

要运行不可执行的sh脚本，请使用：

sh myscript

要运行不可执行的bash脚本，请使用：

bash myscript

启动一个可执行文件（这是任何具有可执行权限的文件）;你只需通过它的路径指定它：

/foo/bar
/bin/bar
./bar

要使脚本可执行，请给予必要的权限：

chmod +x bar
./bar

当一个文件是可执行的，内核负责搞清楚如何执行它。对于非二进制文件，这是通过查看文件的第一行来完成的。它应该包含一个hashbang：

#! /usr/bin/env bash

hashbang告诉内核要运行的程序（在这种情况下，命令/usr/bin/env运行的参数是bash）。然后，脚本被传递给程序（作为第二个参数），以及作为后续参数的所有参数。

这意味着每个可执行的脚本都应该有一个hashbang。如果不是这样，你不会告诉内核它是什么，因此内核不知道用什么程序来解释它。它可能是bash，perl，python，sh，或其他东西。
（实际上，内核通常会使用用户的默认shell来解释这个文件，这是非常危险的，因为它可能根本就不是正确的解释器，或者它可能能够解析一些文件，但是具有微妙的行为差异，比如sh和bash之间的情况）。

A note on /usr/bin/env

最常见的，你会看到像这样的哈希爆炸：

#!/bin/bash

其结果是内核将运行程序/bin/bash来解释脚本。不幸的是，bash并不总是默认发货，并不总是在/bin中可用。在Linux机器上，通常有bash，/usr/xpg/bin/bash或/usr/local/bin/bash等多种POSIX机器。

要编写一个可移植的bash脚本，我们不能依靠硬编码bash程序的位置。
POSIX已经有一个机制来处理这个问题：PATH。这个想法是，你将程序安装在PATH中的一个目录中，当你想运行它时，系统应该能够找到你的程序。

可悲的是，你不能这样做：

#!bash

内核不会（有些可能）为你做一个PATH的搜索。有一个程序可以为你做一个PATH的搜索，但是它叫做env。幸运的是，几乎所有的系统都在env中安装了/usr/bin程序。所以我们使用硬编码路径启动env，然后执行PATH搜索bash并运行它，以便它可以解释您的脚本：

#!/usr/bin/env bash

这种方法有一个缺点：根据POSIX，hashbang可以有一个参数。在这种情况下，我们使用bash作为env程序的参数。这意味着我们没有空间将参数传递给bash。所以没有办法像#!/bin/bash -exu这样的方案转换。您必须在hashbang之后放置set -exu。

这种方法还有另外一个好处：某些系统可能会附带一个/bin/bash，但用户可能不喜欢它，可能会发现它有错误或过时，并可能在其他地方安装了自己的bash。在苹果发布过时的/bin/bash的OS
X（Mac）上，通常情况是这样，用户使用Homebrew等软件安装最新的/usr/local/bin/bash。当你使用env方法进行PATH搜索时，你会考虑用户的偏好，并在他的系统中使用他的首选bash。

要启动shell脚本“file.sh”：

sh file.sh

bash file.sh

另一个选项是使用chmod命令设置可执行权限：

chmod +x file.sh

现在运行.sh文件如下：

./file.sh
